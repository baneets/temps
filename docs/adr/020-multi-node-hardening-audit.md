# Multi-Node Deployment Hardening — Security & Reliability Audit

**Status:** Audit findings (companion to ADR-020)
**Date:** 2026-06-27
**Author:** David Viejo (fan-out audit + analyst verification)
**Scope:** worker enrollment · control-plane↔agent trust boundary · WireGuard mesh · data-plane isolation/routing/DNS · secret distribution · node lifecycle/failover · remote exec · install/join scripts · transport/relay trust · HA Postgres clusters.

---

## How this audit was produced

A fan-out workflow ran **10 independent auditors** (one per dimension), each grounding every finding in real `file:line` code, then **per-finding adversarial verification** (two diverse skeptics each — a *correctness* skeptic told to refute, and an *impact/severity* assessor), then a completeness critic. 161 agents, ~7.9M tokens.

> **⚠️ Caveat — verifier rate-limit.** The session hit its limit late in the run, so the adversarial verifiers for the **secrets**, **lifecycle**, and most **cluster** findings (and the completeness critic) did not finish. To avoid losing signal: the analyst **re-verified the unverified critical/high cluster findings inline** (read `postgres_cluster.rs`, `temps-query-postgres/src/lib.rs`, `service_handlers.rs` directly — all confirmed), and cross-referenced the unverified secrets/lifecycle items against duplicates that *were* confirmed in other dimensions. Items resting only on the finder's word are flagged **🔶 finder-reported, NOT re-verified** and should be confirmed before implementation.

## Severity summary (74 findings; 1 refuted finding excluded)

| Severity | Count |
|---|---|
| 🔴 CRITICAL | 7 |
| 🟠 HIGH | 27 |
| 🟡 MEDIUM | 31 |
| ⚪ LOW | 9 |

The **7 criticals** collapse to three root causes:
1. **The agent is root-Docker-over-plaintext-HTTP behind one static, never-rotated bearer token** (`agentbnd-1/2`, `exec-1/2`, `transport-1/2`). Anyone who reaches the agent port, or who can read cleartext on the path, gets arbitrary `docker exec` on the worker.
2. **The overlay/route/DNS data plane is flat** — blanket nftables `ACCEPT`, every node gets the entire cluster route table + full DNS zone, and the internal proxy will dial any backend (`netiso-1/2`, plus `netiso-3/4`). Zero tenant isolation across the mesh.
3. **HA Postgres clusters ship `0.0.0.0/0 trust` + `host all all 0.0.0.0/0 md5` pg_hba rules, a `LOGIN SUPERUSER` tenant role, and accept-any-certificate TLS** (`cluster-1/2/3/6`), with the data/monitor ports published on `0.0.0.0`.

A fourth structural theme cuts across enrollment: **the cluster join token is a single shared, non-expiring, unlimited-use secret**, and **registration is a name-keyed upsert** — so a leaked join token lets an attacker overwrite any node's identity and redirect every control-plane→agent call (`enroll-1/2`, `agentbnd-4`).

## Agent <-> control-plane trust boundary

> I audited the bidirectional trust boundary between the control plane (CP) and the temps-agent on worker nodes: agent inbound auth (crates/temps-agent/src/auth.rs, server.rs), the CP->agent client (crates/temps-deployer/src/remote.rs), the agent's Docker-driving handlers (service_handlers.rs service_exec, handlers.rs), the long-poll route sync (route_sync_client.rs), node registration/heartbeat on the CP (crates/temps-deployments/src/handlers/nodes.rs + services/node_service.rs), the join/agent CLI provisioning (crates/temps-cli/src/commands/join.rs, agent.rs), and the worker bootstrap (temps-landing/public/worker.sh). Overall posture: the trust boundary is a SINGLE pre-shared bearer token per node, one-directional (the agent never authenticates the CP), with NO mTLS, NO request signing, NO replay protection, and NO per-endpoint authorization. The agent serves plaintext HTTP (axum::serve over a bare TcpListener) and is registered to the CP as an http:// URL, so the token and all deploy/exec/secret payloads cross the wire in cleartext unless an external WireGuard underlay happens to wrap them. The agent process runs as root with the Docker socket and exposes arbitrary command exec in arbitrary containers. There are real strengths (constant-time token compare, strict-by-default TLS in the reqwest clients, SSRF address validation, join-token gating, 0600 config perms), but the core token-only, plaintext, root-Docker design makes any party who can reach the agent port — or who holds the shared join token — able to take over the worker or impersonate it to the CP.

### `agentbnd-1` — 🔴 CRITICAL — Agent drives root Docker with only a shared bearer token, no mTLS / signing / per-endpoint authz

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-agent/src/auth.rs:52-82; crates/temps-agent/src/server.rs:107-109; crates/temps-agent/src/service_handlers.rs:650-710; temps-landing/public/worker.sh:255-266`
- **What's wrong:** The entire agent API is protected by ONE static pre-shared bearer token: `require_agent_auth` (auth.rs:52) checks `Authorization: Bearer <token>` against a single SHA-256 hash and that is the only gate for every route in server.rs (`.layer(middleware::from_fn(require_agent_auth))`, line 107). There is no mTLS, no request signing, no nonce/replay protection, and no per-endpoint authorization — every authenticated caller can hit every endpoint. Those endpoints drive Docker directly: `service_exec` (service_handlers.rs:650) runs `request.command` via `create_exec` in any `request.container_name`, honoring `user: request.user.as_deref()` (so it can request `root`), and `deploy_container`/`remove_container`/`import_image` let the caller create/destroy containers and load arbitrary image tars. The systemd unit installed by worker.sh (`ExecStart=$TEMPS_BIN agent`, no `User=`, `Requires=docker.service`) runs the agent as root with the docker socket, so command execution through the agent is effectively root-on-host.
- **Scenario:** Anyone who learns or captures the node's bearer token (it is transmitted in cleartext, see agentbnd-2, and stored CP-side) — or any process that can reach the agent's TCP port and replay a captured Authorization header — can POST to /agent/services/exec with `command=["sh","-c","..."]`, `user=root`, escaping the container or pivoting to the Docker socket, achieving full root takeover of the worker and every tenant's container/data on it.
- **Recommendation:** Adopt mutual TLS (agent presents a per-node cert, CP presents a client cert the agent pins) so the token is not the sole secret and the channel is authenticated both ways. Add short-lived, signed, per-request capability tokens (or HMAC request signing with a timestamp+nonce) to kill replay. Constrain `service_exec`/`exec_container` to an allowlist of operations and refuse `user: root`/None for untrusted callers. Run the agent as a non-root user in the `docker` group and document the equivalence of docker-socket access to root.

### `agentbnd-2` — 🔴 CRITICAL — Agent serves plaintext HTTP and is registered as http://, so token + deploy/exec/secret payloads cross the wire in cleartext

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-agent/src/server.rs:367-388; crates/temps-cli/src/commands/join.rs:138,288; crates/temps-deployer/src/remote.rs:28-61`
- **What's wrong:** `start_agent_server` binds a bare `tokio::net::TcpListener` and calls `axum::serve(listener, router)` (server.rs:367-388) — there is no TLS termination anywhere in the agent. Consistently, `temps join` registers the agent's address as a plaintext URL: direct mode sends `"address": format!("http://{}:{}", private_address...)` (join.rs:138) and relay mode the same (join.rs:288). The CP builds `RemoteNodeDeployer` from that stored `node.address` (remote.rs:28, agent_url doc-comment even claims https) and `bearer_auth`s every deploy/exec/remove/import/backup call to it. So although `RemoteNodeDeployer` is capable of TLS, in practice the CP talks plain HTTP to the agent. Every request carries the node bearer token in the `Authorization` header and the bodies carry env vars, app secrets, and S3 credentials (ServiceBackupRequest.s3 in lib.rs:204-227).
- **Scenario:** In direct mode (`--private-address` over a user-managed L2/L3 network, the path worker.sh takes by default), there is no transport encryption at all. A passive on-path attacker (compromised switch, shared VLAN, cloud tenant on the same subnet) sniffs the cleartext Authorization header, then replays it to drive root Docker on the worker (agentbnd-1), and simultaneously harvests the S3 credentials and app secrets streamed in deploy/backup bodies — a full data-plane MITM and secret exfiltration.
- **Recommendation:** Terminate TLS on the agent (self-signed per-node cert is fine if the CP pins it) and register the agent as `https://`. Do not rely on the operator happening to deploy WireGuard; the security model must not degrade to plaintext in direct mode. At minimum, refuse to start / loudly warn when the agent is bound to a non-loopback address without TLS.

### `agentbnd-3` — 🟠 HIGH — The agent never authenticates the control plane — CP can be impersonated to push malicious routes/commands

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-agent/src/route_sync_client.rs:140-195; crates/temps-cli/src/commands/agent.rs:39-40,241-249; crates/temps-agent/src/server.rs:160-163`
- **What's wrong:** Trust is one-directional. The agent presents its token TO the CP (route_sync_client.rs:151 `bearer_auth`, heartbeat in server.rs) but does nothing to verify the CP's identity: it accepts whatever `control_plane_url` it was configured with and applies the returned route snapshot unconditionally (`self.store.apply_snapshot(body.generation, routes)`, route_sync_client.rs:188) — the doc-comment even states 'we accept whatever generation the server reports and apply unconditionally'. The CP URL is plain `http://...` in direct mode (the user-supplied join target, join.rs:171 / agent.rs:241) so there is no cert to verify either. The route table the agent ingests directly programs the internal edge proxy's backend addresses (route_store -> internal_proxy).
- **Scenario:** An attacker who can spoof or MITM the agent's CP URL (DNS hijack, ARP/route spoof on the unencrypted direct-mode path, or a rogue host at the configured http endpoint) returns a forged route snapshot pointing victim domains at attacker-controlled backends, silently rerouting internal/preview traffic. Because the agent never validates the CP, there is no detection.
- **Recommendation:** Require the agent to verify the CP via TLS with a pinned CA/cert (mutual TLS) before applying any snapshot. Reject plaintext CP URLs for non-loopback targets. Optionally sign route snapshots with a CP key the agent verifies, so a transport compromise alone cannot forge routes.

### `agentbnd-4` — 🟠 HIGH — Node re-registration by name overwrites token + address using only the shared join token (node hijack / deploy redirect)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-deployments/src/handlers/nodes.rs:573-619; crates/temps-deployments/src/services/node_service.rs:78-99`
- **What's wrong:** `register_node` authorizes registration solely against the cluster-wide `join_token_hash` (nodes.rs:585-619) — a single shared secret, not per-node. `NodeService::register` then looks up any existing node BY NAME and, if found, overwrites its `token_hash`, `token_encrypted`, AND `address`/`private_address` with the attacker-supplied values (node_service.rs:86-98: `active.token_hash = Set(...); active.address = Set(request.address); ...`). The supplied node `token` becomes the new agent credential the CP will use, and `address` becomes the agent URL the CP sends deploy/exec/backup/S3-credential requests to. The only SSRF guard is reserved-range filtering (validate_node_private_address), which still permits any routable public/RFC-1918 IP.
- **Scenario:** An attacker who obtains the shared join token (it is reused for every worker, never rotated per-node, and is passed on CLI/env to every joiner) POSTs /api/internal/nodes/register with `name` of an existing production worker and `address` = attacker host. The CP overwrites that node's address, then routes the next deployment — including injected env secrets and S3 credentials fetched via /s3-credentials — to the attacker's fake agent, and/or the attacker's chosen token now authenticates as that node. This is rogue-node enrollment plus live deploy/secret redirection.
- **Recommendation:** Make the join token single-use or short-TTL and bind it to a specific node identity at issue time. On re-registration, require proof-of-possession of the EXISTING node's token (or its WireGuard key) before allowing token/address overwrite; never let a bearer of the shared join token silently rebind an existing node's address. Audit-log address/token changes on existing nodes.

### `agentbnd-5` — 🟡 MEDIUM — No agent-token rotation or revocation path; compromise is permanent until manual node deletion

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-deployments/src/services/node_service.rs:84-127; crates/temps-cli/src/commands/join.rs:369-374; crates/temps-agent/src/server.rs:36`
- **What's wrong:** The agent token is generated once at join (`generate_token` -> 32 random bytes, join.rs:369), saved to `~/.temps/agent.json`, and loaded for the process lifetime (`AgentAuth::new(&config.token)`, server.rs:36). The CP stores its hash + an encrypted copy and uses it indefinitely. There is no endpoint or service method to rotate or revoke a node's token — `node_service` only updates tokens implicitly via full re-registration (which itself only needs the shared join token, agentbnd-4). The heartbeat/route-sync clients also have no re-key path.
- **Scenario:** After a single token leak (sniffed in cleartext per agentbnd-2, read from a backup of agent.json, or pulled from CP storage), the attacker retains agent access and CP-impersonation-equivalent token indefinitely. The operator's only remedy is to delete and re-create the node, which is disruptive and not surfaced as a security action.
- **Recommendation:** Add explicit per-node token rotation (CP issues a new token, pushes it to the agent over the authenticated channel, invalidates the old hash) and an immediate revocation that the agent honors (e.g. agent re-fetches/validates its credential, or CP marks the token dead so a leaked token stops working without deleting the node). Rotate automatically on a schedule.

### `agentbnd-6` — 🟡 MEDIUM — Agent bind address defaults to 127.0.0.1 but must be manually exposed with no TLS/firewall guidance, inviting a wide-open Docker port

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/agent.rs:215-219; crates/temps-cli/src/commands/join.rs:33-34; crates/temps-agent/src/lib.rs:72-73; temps-landing/public/worker.sh:236-243`
- **What's wrong:** `resolve_config` defaults `listen_address` to `127.0.0.1:3100` (agent.rs:219) and worker.sh joins WITHOUT setting `--agent-address` or `TEMPS_AGENT_ADDRESS`, so the saved config binds loopback. Yet direct-mode join registers the agent at `http://$PRIVATE_IP:3100` (join.rs:138) — an address the CP cannot reach when the agent is on loopback. To make multi-node work the operator must manually rebind the agent to the private IP or 0.0.0.0, but neither worker.sh nor the docs add any firewall rule or TLS. The AgentConfig doc-comment example is literally `"0.0.0.0:3100"` (lib.rs:72). Combined with agentbnd-1/agentbnd-2, an operator following the natural path exposes an unauthenticated-transport, token-only, root-Docker port on a shared network.
- **Scenario:** Operator sees the CP can't reach the worker, rebinds the agent to 0.0.0.0:3100 (as the doc example suggests) to fix it, and adds no firewall. The plaintext Docker-driving port is now reachable by every host on the subnet/VPC; any token capture or token-guess leads to root takeover (agentbnd-1). Reliability angle: the default loopback bind silently breaks CP->agent connectivity (deploys hang as NetworkError) until manually fixed.
- **Recommendation:** Bind the agent to the WireGuard/private interface explicitly (not 0.0.0.0) and reconcile the bind address with the registered address so they cannot diverge. Ship firewall rules in worker.sh that restrict :3100 to the CP/WG source, require TLS when binding non-loopback, and remove the 0.0.0.0 example from the docs.

### `agentbnd-7` — 🟡 MEDIUM — Agent exec/backup handlers act on any container name with no temps-managed ownership check

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-agent/src/service_handlers.rs:650-710,866-1000; crates/temps-agent/src/lib.rs:177-200`
- **What's wrong:** `service_exec` (service_handlers.rs:650) and the backup/restore handlers take an attacker-controllable `container_name` (ServiceExecRequest, lib.rs:177) and call `docker.create_exec(&request.container_name, ...)` / backup commands against it with no verification that the target is a temps-managed container (no `sh.temps.managed` label check, unlike the heartbeat inventory filter in server.rs:183-188). The CP-side exec handler (container_exec.rs) does verify ownership via the DB before delegating, but the agent endpoint itself trusts the caller completely and will exec into ANY container on the host, including non-temps workloads co-located on the box.
- **Scenario:** A caller holding the node token (or replaying a captured request) execs into, dumps, or restores over containers that belong to other software running on the same Docker host (not just temps tenants), broadening blast radius beyond the temps deployment it was scoped to and enabling cross-workload compromise / data exfiltration on shared hosts.
- **Recommendation:** On the agent, enforce that exec/backup/restore targets carry the `sh.temps.managed=true` label (and, where applicable, the expected project/env labels) before running any Docker operation, mirroring the inventory filter already used in the heartbeat path. Reject operations against unlabeled/foreign containers.

**✅ Strengths already in place (do NOT regress):**

- Token comparison is hardened: `AgentAuth::verify` SHA-256-hashes both sides and uses a constant-time byte compare (auth.rs:25-49); the CP-side heartbeat/S3/join-token checks reuse the same constant_time_eq (nodes.rs:485-494, 591, 816, 911).
- The reqwest clients are strict-TLS BY DEFAULT and the code documents the threat: the heartbeat client has no insecure opt-in at all (server.rs:148-158), the join clients always validate certs and the comments explicitly explain the token-theft/MITM risk (join.rs:124-130, 264-270), and RemoteNodeDeployer only relaxes verification behind an explicit operator-set `insecure_tls_enabled()` global (remote.rs:42-53).
- Node registration is gated: registration is refused entirely unless a join token is configured, and a missing/invalid join token is rejected with FORBIDDEN (nodes.rs:585-619). The agent token is hashed before DB storage and also stored encrypted via EncryptionService (nodes.rs:621,660-668).
- SSRF hardening on registration: both `private_address` and the `address`/agent-URL host are validated against loopback, link-local/cloud-metadata, unspecified, multicast, broadcast, and documentation ranges before being persisted and later used to build agent/health URLs (nodes.rs:379-538, 626-656).
- Edge TLS private keys are never sent to plain workers — cert distribution is role-gated (`if node.role != "edge"`) and ECIES-encrypted to the node's registered X25519 public key with per-sync ephemeral keys (nodes.rs:1183-1271).
- The token CLI arg is `hide = true` and env-var-preferred to avoid process-list exposure (agent.rs:30-32), and `temps join` writes agent.json / its directory with 0600 / 0700 permissions (join.rs:86-102).
- Heartbeat reliability is well-built: bounded exponential backoff, retry only on network/5xx, first-beat container inventory for CP-side reconciliation of stale records, and clear escalation logging (server.rs:118-309).

## Data-plane network isolation, routing & DNS sync

> I audited the cross-node data plane: the worker internal proxy (temps-agent/internal_proxy.rs), route mirroring (route_store.rs, route_sync_client.rs) and its CP source (temps-routes/route_sync.rs + route_table.rs), DNS sync (temps-dns-resolver/sync_client.rs + temps-dns/handlers/dns_sync.rs + services/dns_registry.rs), the overlay network (temps-network: config.rs, allocator.rs, linux/firewall.rs), WireGuard relay (temps-wireguard/lib.rs), and the agent HTTP surface (server.rs, auth.rs). The control/agent plane has a coherent per-node bearer-token auth model with constant-time comparison and strict TLS on worker→CP calls, and CIDR allocation is race-safe via a partial-unique index. The data plane, however, is a single flat overlay with NO tenant segmentation, and the route/DNS sync endpoints ship the ENTIRE cluster-wide table to every node rather than scoping to the node's own containers — so any worker (or any container that can reach the bridge) can both enumerate and reach every other tenant's internal services across the mesh. The internal proxy also lacks any backend allow-listing or inbound forwarded-header sanitization. Overall posture: control-plane auth is solid; data-plane isolation between tenants is effectively absent.

### `netiso-1` — 🔴 CRITICAL — Flat overlay with blanket nftables ACCEPT — zero tenant/container isolation across the mesh

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-network/src/linux/firewall.rs:70-72 (render_baseline); crates/temps-network/src/config.rs:104-110 (single VNI 42, one bridge)`
- **What's wrong:** The baseline firewall installs `add chain ... forward { ... policy accept; }` plus `forward iifname "br-temps0" accept` and `forward oifname "br-temps0" accept` — every packet ingressing or egressing the overlay bridge is accepted unconditionally. All nodes share one VNI (`Transport::Vxlan { vni: 42 }`) and one bridge (`br-temps0`/`temps0`), and every container from every project/tenant is attached to that same L3 segment. There is no per-tenant CIDR partition, no per-container ACL, no source/destination filtering — `validate_with` only checks that node CIDRs don't *overlap*, not that tenants are separated.
- **Scenario:** Tenant A deploys a container on node 1 (172.20.5.0/24). Tenant B's container on node 2 (172.20.6.0/24) is on the same overlay. B's container runs `curl http://172.20.5.7:5432` (or any port) and reaches A's Postgres/Redis/app directly — the VXLAN carries the frame, the FORWARD chain accepts it, and there is no rule that says "B may not talk to A". One compromised or malicious customer container pivots laterally into every other customer's databases and internal apps cluster-wide.
- **Recommendation:** Segment the data plane per tenant/project: either (a) one overlay network (distinct VNI + bridge) per tenant, or (b) nftables rules that only permit container↔gateway and intra-project flows and DROP inter-CIDR container-to-container traffic by default (allow-list the specific service dependencies a project declares). At minimum replace `policy accept` with `policy drop` and add explicit ACCEPT rules scoped to (a) the bridge gateway and (b) same-project peers.

### `netiso-2` — 🔴 CRITICAL — Route snapshot endpoint returns the entire cluster route table to every node (cross-tenant route disclosure + reach)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-routes/src/route_sync.rs:111-145 (get_routes_snapshot) + 219-251 (build_snapshot / snapshot_internal_routes); consumed by crates/temps-agent/src/route_store.rs:83-115`
- **What's wrong:** `get_routes_snapshot` authenticates the node, then returns `build_snapshot(&app_state.peer_table, current)` which calls `peer_table.snapshot_internal_routes()` — every `*.temps.local` route in the whole cluster, with backend addresses, deployment_id, project_id, environment_id. The path `node_id` is validated against the token but is NEVER used to filter the routes. The doc comment claims "a worker cannot fetch another worker's state," but the snapshot is identical for all nodes. The agent's RouteStore then serves all of them via the internal proxy, which dials `backend.address` as-is (internal_proxy.rs:522 `format!("http://{}{}", backend.address, ...)`).
- **Scenario:** Any registered worker (or an attacker who obtained one node token) polls `/api/internal/nodes/{id}/routes/snapshot` and receives the full internal topology of every tenant — backend overlay IPs, ports, project/deployment IDs — and, because the proxy will dial any address in its store, can proxy requests to any tenant's internal backend by sending the right `Host:` header to its local bridge proxy.
- **Recommendation:** Scope the snapshot to routes whose backends actually live on (or are legitimately reachable by) the requesting node, or that belong to projects the node is authorized to host. Filter `snapshot_internal_routes()` by node ownership / project assignment before returning. Do not leak project_id/deployment_id of other tenants.

### `netiso-3` — 🟠 HIGH — DNS sync endpoint returns the full *.temps.local zone (all tenants) to any authenticated node

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-dns/src/services/dns_registry.rs:532-599 (get_changes_since / get_full_zone — filter only by Generation, never node_id); served at crates/temps-dns/src/handlers/dns_sync.rs:148-172`
- **What's wrong:** `get_changes_since(since)` and `get_full_zone()` query `service_endpoints` filtered solely on `Generation.gt(since)` (or unfiltered for a full snapshot) — there is no `node_id`/owner filter. `get_dns_changes` authenticates the node but passes only `q.since`. Every per-node Hickory resolver therefore learns every tenant's internal FQDN → IP:port mapping. The handler comment asserts "a worker cannot fetch another worker's state," which is false for the record set.
- **Scenario:** A worker (or holder of a single node token) calls `/api/internal/nodes/{id}/dns/changes?since=0` and enumerates the entire internal service map of the cluster — every project's `<env>.<project>.temps.local` and service-member records with target IPs/ports — providing a precise target list for the lateral movement enabled by finding netiso-1.
- **Recommendation:** Scope DNS changes to records the requesting node needs to resolve (records for projects scheduled on that node, plus shared infrastructure it is entitled to). Add a node/owner filter to `get_changes_since`/`get_full_zone`, or partition zones per tenant so a node only mirrors its own tenants' records.

### `netiso-4` — 🟠 HIGH — Internal proxy dials any backend address with no allow-list, plus client-controllable Host selects the route (SSRF / cross-tenant proxying)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-agent/src/internal_proxy.rs:129-176 (handle), 509-522 (forward_once builds upstream_url from backend.address), 199-234 (proxy_upgrade dials backend.address)`
- **What's wrong:** The proxy accepts any request on `<bridge_ip>:80` from any container on the overlay, extracts the `Host:` header, and forwards to whatever `backend.address` the (cluster-wide, unscoped — see netiso-2) route store holds. The only guard is `host.ends_with(".temps.local")`. There is no verification that the calling container is entitled to the matched route, and `backend.address` is dialed verbatim (`http://{addr}{path}`) — for cross-node routes this is an underlay IP:port, turning the proxy into a confused deputy that can reach other nodes' published ports.
- **Scenario:** Tenant B's container resolves the bridge IP and sends `GET / HTTP/1.1` with `Host: prod.tenant-a.temps.local`. The proxy matches A's route in its (cluster-wide) store and proxies B's request straight into A's backend, including across nodes via the dial-as-is underlay address — cross-tenant data access without ever touching A's overlay CIDR directly.
- **Recommendation:** Authorize the caller against the matched route (e.g. by mapping the source overlay IP to its owning project/container and checking it matches the route's project, or by only serving routes the node owns per netiso-2). Constrain `backend.address` to an allow-list of known overlay/underlay endpoints and reject addresses outside the expected ranges.

### `netiso-5` — 🟡 MEDIUM — Inbound x-forwarded-* and x-temps-deployment-id headers are forwarded (not stripped), so any overlay client can spoof them to backends

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-agent/src/internal_proxy.rs:536-552 (forward_once copies all non-hop-by-hop headers, then appends its own) and 240-269 (proxy_upgrade)`
- **What's wrong:** `forward_once` copies every client header except the HOP_BY_HOP set, then calls `builder.header("x-forwarded-proto", ...)`, `builder.header("x-forwarded-host", ...)`, and `builder.header("x-temps-deployment-id", ...)`. reqwest's `.header()` APPENDS rather than replaces, and inbound `x-forwarded-*`/`x-temps-deployment-id` are never removed before copy. The proxy module doc claims it 'adds' these headers, but it never sanitizes the client-supplied versions. The same applies to the raw upgrade path which only drops a narrow hop-by-hop subset.
- **Scenario:** A container sends `X-Forwarded-Host: admin.victim.temps.local` and `X-Temps-Deployment-Id: 999` to the bridge proxy. Both reach the backend (duplicated alongside the proxy's values). A backend that trusts the first occurrence of X-Forwarded-Host (common) makes a trust/authorization decision on attacker-controlled input, or the spoofed deployment-id pollutes log-correlation/audit.
- **Recommendation:** Strip inbound `x-forwarded-for`, `x-forwarded-host`, `x-forwarded-proto`, and `x-temps-deployment-id` from the request before copying headers, then set them authoritatively. Use a replace/insert (not append) semantic so only the proxy's values are present.

### `netiso-6` — 🟡 MEDIUM — Per-node auth accepts a deleted/decommissioned node's token (no status check); single static node token never rotated

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-routes/src/route_sync.rs:259-289 (authenticate_node); crates/temps-dns/src/handlers/dns_sync.rs:230-258; crates/temps-deployments/src/handlers/network.rs:110-128; crates/temps-agent/src/auth.rs:14-31`
- **What's wrong:** Every node-facing endpoint authenticates by looking up the node row and constant-time comparing `node.token_hash`. None of them check a node status/enabled/revoked flag — if a node is being decommissioned but the row still exists, its token still grants full route+DNS+peer access. The token itself is a single long-lived static secret (`AgentConfig.token`, set once at join, `request.token` supplied at registration) used for the agent API, heartbeat, route sync, DNS sync and peer list, with no rotation or per-purpose scoping path in this subsystem. Compromise of one node's token yields the cluster-wide route/DNS disclosure of netiso-2/netiso-3 until the row is manually deleted.
- **Scenario:** An operator marks a worker offline/quarantined but the node row lingers; the (possibly leaked) token continues to authenticate route/DNS/peer pulls, leaking the whole internal topology. There is also no way to rotate a suspected-leaked node token short of deleting and re-registering the node.
- **Recommendation:** Add an explicit node status check (active/enabled) to every `authenticate_node` path and reject tokens for non-active nodes. Provide a token-rotation endpoint and treat node tokens as rotatable credentials; consider short-lived tokens or per-endpoint scoping.

### `netiso-7` — 🟡 MEDIUM — Route/DNS apply is fully trusting and unauthenticated-at-the-payload-level — a MITM or compromised CP path can inject hijack routes the agent applies blindly

- **Verification:** ❌ refuted by verifiers
- **Category:** both · **Confidence:** medium · **Location:** `crates/temps-agent/src/route_sync_client.rs:140-195 (apply_snapshot unconditionally) + crates/temps-agent/src/route_store.rs:83-109; crates/temps-dns-resolver/src/sync_client.rs:106-162 (replace/apply_diff)`
- **What's wrong:** The agent applies whatever the snapshot/changes response contains with no integrity verification beyond TLS transport and no sanity bounds: `apply_snapshot` replaces the entire host→backend map, and the DNS client `replace`s the whole zone. The route_sync_client (unlike network_sync, which sets `danger_accept_invalid_certs(false)`) builds its reqwest client at route_sync_client.rs:93-95 with only a timeout and no explicit TLS-strictness assertion, relying on reqwest defaults. There is no payload signing, so trust rests entirely on the TLS channel and the CP. A route entry pointing `victim.temps.local` at an attacker backend would be applied verbatim and served by the proxy.
- **Scenario:** If TLS to the CP is downgraded/misconfigured (or the CP route table is poisoned upstream — e.g. via a route-injection bug), the agent installs hijacked `*.temps.local`→attacker-IP routes and DNS records on every node, silently MITMing internal service-to-service traffic. There is no agent-side validation that a route's backend lies within an expected/owned CIDR.
- **Recommendation:** Pin/assert strict TLS explicitly in the route sync client (mirror network_sync). Add agent-side validation that backend addresses fall within known overlay/underlay CIDRs and that hosts match expected ownership before applying. Consider signing the snapshot payload (CP-held key) so a transport compromise alone cannot inject routes.

### `netiso-8` — ⚪ LOW — Stale-snapshot serving on CP restart and best-effort ACK loss can leave nodes serving wrong/old routes with no drift enforcement

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** reliability · **Confidence:** medium · **Location:** `crates/temps-agent/src/route_sync_client.rs:9-24 + 164-189 (apply when generation differs OR store empty; ACK best-effort .ok()); crates/temps-routes/src/route_sync.rs:188-193 (ack persistence logged-only); crates/temps-agent/src/route_store.rs:129-168 (load_from_disk serves stale on cold start)`
- **What's wrong:** On CP restart the in-memory generation resets to a low number; the client comment acknowledges `since` may exceed server `current` and it 'applies unconditionally,' but the apply only fires when `body.generation != since || store.is_empty()`. If the CP comes back at a *lower* generation than what the node already applied with a *changed* route set at that same number, the equality check can mask a needed update, and the route ACK barrier (`post_routes_ack`) is best-effort (persistence failure is only logged), so `mark_deployment_complete`'s 'all workers applied' barrier can be satisfied or stalled incorrectly. Cold-start serves disk snapshot (route_store load_from_disk) which may be arbitrarily stale until first successful sync.
- **Scenario:** CP restarts and its generation counter resets; a node that had applied generation N keeps serving its old map because the new server generation collides numerically, routing internal traffic to backends that have since moved — an internal partial outage that self-heals only on the next true generation bump. ACK persistence failures additionally desync the CP's view of which nodes are current.
- **Recommendation:** Make generations monotonic across CP restarts (persist the counter) or have the agent reconcile by content hash, not just the numeric generation. Treat a numerically-lower server generation as 'force full re-sync.' Make the route ACK barrier surface persistence failures instead of silently logging, so the deployment barrier reflects true worker state.

**✅ Strengths already in place (do NOT regress):**

- Per-node bearer-token auth is consistent across route sync, DNS sync, peer list, and heartbeat, with SHA-256 hashing and constant-time comparison (temps-agent/src/auth.rs:40-49; route_sync.rs constant_time_eq; dns_sync.rs constant_time_eq; network.rs constant_time_eq) — no plaintext token storage, no naive == comparison.
- Worker→CP calls that carry the node token enforce strict TLS explicitly: network_sync.rs:113-117 sets danger_accept_invalid_certs(false) with a comment explaining the MITM/token-capture risk; heartbeat loop (server.rs:145-158) documents the same and uses default-strict reqwest.
- Node registration has an SSRF guard: register_node validates both private_address and the agent `address` host via validate_node_private_address, rejecting reserved/dangerous ranges before they become health-check/deploy URLs (nodes.rs:623-656). Node enrollment is gated by a configured join-token hash and refused entirely when multi-node is not enabled.
- CIDR allocation is race-safe and well-reasoned: PostgresAllocator allocates inside a transaction, picks the lowest free non-overlapping subnet, and relies on a partial-unique index as the concurrency backstop (allocator.rs:148-268); malformed persisted CIDRs are surfaced loudly rather than silently shadowing a free subnet.
- The internal proxy correctly restricts itself: binds to `<bridge_ip>:80` only (never 0.0.0.0, never Docker-published), refuses non-`*.temps.local` hosts, strips RFC 7230 hop-by-hop headers, disables redirect-following (redirect::Policy::none) to avoid SSRF-via-redirect, and bounds upstream timeouts/retries (internal_proxy.rs:40-43, 105-112, 142-147).
- Route store uses atomic whole-map replacement plus temp-file+rename for crash-safe disk snapshots, and the sync loops have bounded exponential backoff with cooperative shutdown — no busy-spin on CP outage (route_store.rs:170-183; route_sync_client.rs:107-138).
- DNS resolver binds only to the bridge gateway and 127.0.0.53 (not 0.0.0.0), and the upstream forwarder does not cache (preserving short failover TTLs), passes NXDOMAIN through cleanly, and never panics or leaks anyhow into the wire (config.rs:75-77; upstream.rs:16-21,103-128).
- DNS ACK validation rejects an agent ACKing a generation higher than the server ever issued (AckTooHigh), preventing an agent from advancing the CP's applied-generation view beyond reality (dns_registry.rs:274-276, 605-611).

## Install/join/worker scripts & supply chain

> I audited the node-bootstrap supply chain for temps.sh: the canonical curl|bash scripts served from temps-landing/public/ (install.sh, worker.sh, deploy.sh, quick.sh, upgrade.sh), the in-repo scripts/install.sh and scripts/release.sh, the GitHub Actions release workflow (.github/workflows/release.yml), the systemd units they write, the Rust join path (crates/temps-cli/src/commands/join.rs), and the Rust self-upgrade path (crates/temps-cli/src/commands/upgrade.rs) for contrast. The release pipeline DOES publish a SHA256 checksum next to every tarball, and the Rust `temps upgrade` command DOES download and verify it — but every bash bootstrap script that performs the FIRST install of the binary on a worker/control-plane node (the most security-critical moment) downloads the tarball with no integrity check whatsoever. The systemd units written by the worker and control-plane installers run with broad privileges and embed the database password on the ExecStart command line. The join token is conveyed as a plaintext positional CLI argument. Overall posture: artifact-integrity material exists but is unused on the install path, and the host-hardening of the resulting services is minimal.

### `supplychain-1` — 🔴 CRITICAL — Worker/control-plane install scripts download the binary with no checksum or signature verification despite published .sha256 assets

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `temps-landing/public/worker.sh:187-191; temps-landing/public/install.sh:221-228; scripts/install.sh:194-199; temps-landing/public/upgrade.sh:160-163`
- **What's wrong:** Every first-install path downloads and extracts the binary with zero integrity verification. worker.sh: `curl --fail --location --progress-bar --output "$exe.tar.gz" "$url"` then `tar -xzf "$exe.tar.gz"` then `chmod +x "$exe"` — no sha256 step. install.sh, scripts/install.sh and upgrade.sh are identical in this respect. A grep for `sha256|shasum|checksum|gpg|cosign|sigstore` across all public scripts returns 0 matches. This is despite the release workflow PUBLISHING a checksum for every artifact: `.github/workflows/release.yml:97 sha256sum temps-linux-amd64.tar.gz > temps-linux-amd64.tar.gz.sha256` (and uploading `temps-linux-amd64.tar.gz.sha256` at line 111). The integrity material exists and is simply ignored on the install path. By contrast the Rust `temps upgrade` DOES verify it (crates/temps-cli/src/commands/upgrade.rs:311-318 `verify_checksum(&tarball_bytes, &checksum_text)?`), proving the omission in the shell installers is an inconsistency, not a missing capability.
- **Scenario:** A network MitM, a compromised GitHub release asset, a hijacked github.com/codeload CDN edge, or a malicious mirror via the `GITHUB` override env var (`GITHUB=${GITHUB-"https://github.com"}`, install.sh:118) serves a trojaned tarball. The worker node — which is a Docker host that the control plane will instruct to run arbitrary containers — executes the attacker's binary as a systemd service on first boot. This is full remote node takeover at enrollment time, the highest-impact moment in the lifecycle.
- **Recommendation:** Download the corresponding `<tarball>.sha256` from the same release, compute `sha256sum`/`shasum -a 256` locally, and abort the install on mismatch in all four scripts. Treat a MISSING checksum asset as a hard failure (do not silently skip). For stronger guarantees, sign releases (cosign/sigstore or minisign) and ship the public key inside the script so the signature — not just an attacker-replaceable sidecar hash — is verified.

### `supplychain-2` — 🟠 HIGH — systemd unit embeds the database password (with credentials) on the ExecStart command line

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `temps-landing/public/deploy.sh:2197-2202 and :2971-2977`
- **What's wrong:** Both the advanced and quick control-plane installers write the DB connection string — which contains the Postgres password (`DB_PASSWORD`, built at deploy.sh:754/883/913) — directly into the systemd ExecStart: `ExecStart=$temps_bin serve ... --database-url="$db_url" ...`. The unit file at `/etc/systemd/system/temps.service` is world-readable by default, and the full argv (including `--database-url=postgres://temps:<password>@...`) is visible to any local user via `ps aux`, `/proc/<pid>/cmdline`, and `systemctl cat temps`.
- **Scenario:** Any unprivileged local process or user on the control-plane host reads the Postgres superuser-equivalent password from `ps`/`/proc` or the world-readable unit file, then connects directly to TimescaleDB and reads/exfiltrates all tenant data, secrets, session tokens, and encryption material stored in the database.
- **Recommendation:** Pass the database URL via `Environment=` from a `0600` root-owned EnvironmentFile (systemd `EnvironmentFile=/etc/temps/temps.env`) or via `TEMPS_DATABASE_URL`, never as a command-line argument. Ensure the unit file itself is `chmod 600`. Verify the binary reads the URL from env and does not echo it in logs.

### `supplychain-3` — 🟠 HIGH — Generated systemd units have no privilege/sandbox hardening (no NoNewPrivileges, ProtectSystem, ProtectHome, etc.)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `temps-landing/public/worker.sh:255-271; temps-landing/public/deploy.sh:2187-2211 and :2961-2986`
- **What's wrong:** The temps-agent unit (worker.sh:255-271) and the temps.service control-plane units (deploy.sh) contain only `Type/User/ExecStart/Restart/RestartSec/Environment` (plus `LimitNOFILE`/`AmbientCapabilities=CAP_NET_BIND_SERVICE` on the server). A grep for `NoNewPrivileges|ProtectSystem|ProtectHome|PrivateTmp|ProtectKernel*|RestrictNamespaces|CapabilityBoundingSet` across deploy.sh and quick.sh returns 0. The worker agent unit has no `User=` at all, so when worker.sh runs as root (the documented `curl | bash` case where `$EUID==0`, worker.sh:115-117) the agent runs as full root with no sandboxing. The control-plane unit at least sets `User=$RUN_USER`, but `RUN_USER="$(whoami)"` (deploy.sh:99) — i.e. whoever ran the installer, often root.
- **Scenario:** A code-exec or container-escape bug in the agent or server (the agent is internet-reachable and the server proxies untrusted deployed apps) immediately yields the unit's full privilege set. With no `NoNewPrivileges` a sub-process can re-acquire setuid privileges, and with no `ProtectSystem=strict`/`ProtectHome` it can tamper with `/etc`, other users' homes, and the binary itself — turning a single bug into persistent host compromise.
- **Recommendation:** Add the standard hardening block to both units: `NoNewPrivileges=true`, `ProtectSystem=strict`, `ProtectHome=read-only` (or carve out the data dir via `ReadWritePaths=`), `PrivateTmp=true`, `ProtectKernelTunables=true`, `ProtectKernelModules=true`, `RestrictNamespaces=` as feasible (note the agent needs Docker socket access, so scope carefully), and a dedicated non-login service account (`User=temps`) rather than `whoami`/root.

### `supplychain-4` — 🟠 HIGH — Worker piped-curl into get.docker.com with no verification, then runs the agent as root

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `temps-landing/public/worker.sh:145`
- **What's wrong:** worker.sh installs Docker via `curl -fsSL https://get.docker.com | $SUDO sh >/dev/null 2>&1` — a remote script piped straight into a root shell with no pinning, no checksum, and stdout/stderr discarded so failures are masked behind a generic `Docker installation failed`. This is a second, fully unverified supply-chain entry point in the same bootstrap that established the (already unverified) Temps binary. The resulting Docker daemon plus the temps-agent both run as root on the node.
- **Scenario:** A compromise or MitM of get.docker.com (or its CDN) during enrollment installs a backdoored container runtime on the worker as root. Because output is suppressed, a partial or substituted install can also slip past unnoticed. The node is a Docker host the control plane drives, so this is host takeover.
- **Recommendation:** Pin to a specific Docker install method (distro packages from the official Docker apt/yum repo with GPG-verified package signatures, or a pinned get.docker.com version with a checksum). Do not discard installer output; surface real errors. Document the trust assumption so operators can pre-provision Docker out-of-band.

### `supplychain-5` — 🟠 HIGH — Join token is passed as a plaintext positional CLI argument (leaks via ps, shell history, and process listing)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `temps-landing/public/worker.sh:25-28,84,237-243; crates/temps-cli/src/commands/join.rs:137`
- **What's wrong:** The documented enrollment is `curl -fsSL https://temps.sh/worker.sh | bash -s -- <CONTROL_PLANE_URL> <JOIN_TOKEN>` (worker.sh:25-28); the token is captured as `JOIN_TOKEN="${2:-}"` (worker.sh:84) and re-passed verbatim to `temps join "$CONTROL_PLANE_URL" "$JOIN_TOKEN" ...` (worker.sh:237-243). A plaintext token on argv is visible to every local user via `ps aux`/`/proc/<pid>/cmdline` for the lifetime of the process, is recorded in shell history when an operator runs it interactively, and may land in command-logging/auditd. The token is the credential that authorizes registering a new worker into the cluster (join.rs:137 `"join_token": self.token`).
- **Scenario:** A co-tenant or low-priv local process on the worker (or a leaked shell-history/CI log) captures the join token while enrollment runs. If the token is reusable/long-lived (see supplychain-6), the attacker registers their own rogue worker node into the victim's cluster and receives container deploy/exec instructions, or impersonates the node.
- **Recommendation:** Accept the join token via an environment variable or a `0600` file path (`--join-token-file`) read by `temps join`, not as a positional argv. In worker.sh, read it from stdin or `TEMPS_JOIN_TOKEN` env rather than `$2`. Ensure the token is single-use and short-TTL (see supplychain-6).

### `supplychain-6` — 🟡 MEDIUM — Installer trusts unauthenticated GitHub API ordering for 'newest' release with no draft/signature filtering (beta channel and resolution)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** medium · **Location:** `temps-landing/public/install.sh:133-143; temps-landing/public/worker.sh:172-181; scripts/install.sh:144-157`
- **What's wrong:** Version resolution greps the first `"tag_name":` out of an unauthenticated `api.github.com/.../releases?per_page=20` (beta) or `/releases/latest` (stable) response and uses it verbatim: install.sh:135-141 `ver=$(curl ... | grep '"tag_name":' | head -1 | cut -d'"' -f4)`. The script's own comments (scripts/install.sh:138-143) acknowledge it does NOT filter drafts and 'trusts that the API only returns shipped releases' — explicitly weaker than the Rust CLI which checks `draft`. There is no pinning of the expected version range and no signature, so whatever the API returns first becomes the installed version.
- **Scenario:** If an attacker can publish or surface a malicious/rolled-back release (compromised maintainer token, or a draft that becomes visible), or MitM the unauthenticated API JSON, the beta worker fleet auto-resolves to the attacker-chosen tag and (combined with supplychain-1's lack of checksum) installs it. Even absent compromise, a botched/yanked release can be silently picked up across all new nodes.
- **Recommendation:** Verify the resolved tag against a signed release manifest, or require an explicit pinned version for fleet/worker installs. Filter drafts/prereleases properly (parse JSON, e.g. with a guaranteed parser) rather than `grep | head -1`. Combined with supplychain-1, verify the artifact signature so a substituted tag cannot also substitute the binary.

### `supplychain-7` — 🟡 MEDIUM — worker.sh accepts any CONTROL_PLANE_URL scheme (no https enforcement) for enrollment

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** medium · **Location:** `temps-landing/public/worker.sh:83,237-243`
- **What's wrong:** `CONTROL_PLANE_URL="${1:-}"` (worker.sh:83) is passed straight to `temps join "$CONTROL_PLANE_URL" ...` with no scheme validation; the script only checks the value is non-empty. The Rust join_direct path builds `register_url = format!("{}/api/internal/nodes/register", self.target)` from this URL verbatim (join.rs:132). join.rs:124-130 documents that direct mode 'may traverse the public internet' and 'always require[s] valid TLS' — but that protection only holds if the operator types an https:// URL. A plain http:// (or typo'd) URL would send the join token in cleartext.
- **Scenario:** An operator (or a copy-paste from a doc/dashboard) supplies an `http://` control-plane URL. The join token (which also flows in the register body, join.rs:137) and the generated agent token are transmitted in cleartext and can be sniffed, letting an attacker register a rogue worker or hijack the node identity.
- **Recommendation:** Reject non-https CONTROL_PLANE_URLs in worker.sh (allow http only for explicit localhost/private-IP dev), and enforce the same scheme check inside `temps join` so the CLI refuses to send a join token over plaintext regardless of how it is invoked.

### `supplychain-8` — 🟡 MEDIUM — Rust self-upgrade silently proceeds when no checksum asset is present (downgrade-to-no-verify)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/upgrade.rs:313-322`
- **What's wrong:** The otherwise-correct verification logic finds the checksum by name and only verifies if present: `let checksum_asset = release.assets.iter().find(|a| a.name == checksum_name); if let Some(checksum_asset) = checksum_asset { ... verify_checksum(...)?; } else { debug!("No checksum asset found, skipping verification"); }`. A missing or renamed `.sha256` asset causes the upgrade to install the tarball with no integrity check, and the skip is only a `debug!` log so operators won't notice.
- **Scenario:** An attacker who can influence the release asset list (or a release published without its checksum) causes `temps upgrade` to fetch and run an unverified binary across the fleet — silently bypassing the very protection this code path implements.
- **Recommendation:** Make a missing checksum a hard error (`bail!`) unless an explicit `--insecure`/`--no-verify` flag is passed. Log the skip at `warn!`/`error!`, not `debug!`. Longer term, require a signature so the check cannot be defeated by simply omitting the sidecar hash.

**✅ Strengths already in place (do NOT regress):**

- The release pipeline already generates and publishes a SHA256 checksum for every artifact on every platform (.github/workflows/release.yml:97-111, 181-195, 263-277, 345-...), so the integrity material needed to harden the installers already exists — only the consumption is missing.
- The Rust `temps upgrade` self-update path correctly downloads the .sha256 asset and verifies it before installing (crates/temps-cli/src/commands/upgrade.rs:311-318, verify_checksum at :737-748 using sha2::Sha256), and the EE path gates the checksum behind a license proxy (:457-463). This is the right pattern to port to the bash installers.
- The agent config file written by `temps join` (containing the agent token and control-plane URL) is chmod 0600 on Unix (crates/temps-cli/src/commands/join.rs:100-101), so the persisted node credential is not world-readable.
- join.rs explicitly refuses the server-side `insecure_tls` opt-in for the CLI register call and documents the MitM/token-theft threat (join.rs:124-130), requiring valid TLS for the registration request itself (the gap is only that the URL scheme isn't enforced — supplychain-7).
- deploy.sh chmods sensitive state and cert material to 600 (deploy.sh:448 state dir files, :1383 cert PEMs), showing the author already applies least-privilege file modes in places — the systemd/db-url paths just weren't held to the same standard.
- Release channel selection is CLI-flag-only with no env-var fallback (install.sh:81-88, worker.sh:55-60), deliberately preventing a stray CI/shell env var from silently switching a host onto the beta channel.
- Both install.sh and worker.sh wrap execution in a `{ ... }` block and a POSIX bash-guard re-exec, so a truncated curl|bash download will not execute a partial script.

## HA service cluster membership & failover integrity

> I audited the multi-node Postgres HA cluster path (pg_auto_failover): the entity (crates/temps-entities/src/service_members.rs), the cluster provider (crates/temps-providers/src/externalsvc/postgres_cluster.rs), the role/VIP reconciler (postgres_role_reconciler.rs), membership lifecycle in services.rs (init/add/remove/promote cluster member), the HTTP handlers (handlers/handlers.rs), the worker agent that creates containers (temps-agent/src/service_handlers.rs + auth.rs), and the monitor/health TLS connection (temps-query-postgres/src/lib.rs).  Posture: the *control-plane API surface* is well-guarded — every cluster handler enforces `permission_guard!(auth, ExternalServices*)` plus `assert_service_owned_by_caller`, write ops are audited, the worker agent requires a constant-time-compared SHA-256 bearer token, ordinal allocation is race-safe behind a unique constraint with bounded retry, and remove/promote enforce quorum + live-primary guards. The serious weaknesses are all in the *cluster data plane*: the pg_auto_failover containers are configured with blanket `0.0.0.0/0 trust` (monitor) and `host all all 0.0.0.0/0 md5` (data nodes) in pg_hba.conf, their ports are bound to `0.0.0.0` on the worker, and every control-plane→member TLS connection accepts ANY certificate (and silently falls back to plaintext). Together these create cross-tenant takeover and MITM exposure for anyone with underlay/overlay network reach. A reliability gap exists in the monitor-unreachable "live primary" fallback.

### `cluster-1` — 🔴 CRITICAL — pg_auto_failover monitor accepts autoctl_node with trust auth from 0.0.0.0/0 (no password) — cluster control-plane takeover

- **Verification:** ✅ verified inline by analyst (verifier rate-limited)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-providers/src/externalsvc/postgres_cluster.rs:176-181, monitor_command()`
- **What's wrong:** The monitor entrypoint unconditionally appends `echo 'hostssl pg_auto_failover autoctl_node 0.0.0.0/0 trust'` (and the `::/0` variant) to pg_hba.conf, and the monitor port is published on the worker host (agent binds `host_ip: "0.0.0.0"`). `trust` means NO password — any party who can open a TLS socket to the monitor port can connect as `autoctl_node` to the `pg_auto_failover` database. That role owns the formation: it can register rogue nodes, `pg_autoctl drop node`, and drive `pgautofailover.node` state — i.e. force failovers or wedge the FSM. Confirmed by probe_cluster() at services.rs:2236-2244 which connects with `user=autoctl_node ... sslmode=require` and an empty password.
- **Scenario:** An attacker (another tenant's container on the temps overlay, a compromised neighbouring worker, or anyone who reaches the worker's private/WG IP and the 6000+service_id*10 port) connects to the monitor over TLS as autoctl_node with no credentials, then issues monitor commands to drop the real primary/replicas or register a malicious node — taking over or destroying the HA cluster of an unrelated tenant.
- **Recommendation:** Replace the global `0.0.0.0/0 trust` with (a) certificate auth for autoctl_node (pg_auto_failover supports `--ssl-ca-file` / cert-based registration), or at minimum a password and an hba scoped to the cluster's own overlay subnet (e.g. the `172.20.0.0/16` overlay CIDR), never `0.0.0.0/0`. Bind the monitor port to the overlay/WireGuard interface, not `0.0.0.0`. Enforce a host firewall on the cluster port range.

### `cluster-2` — 🔴 CRITICAL — Data-node pg_hba.conf opens `host all all 0.0.0.0/0 md5` + replication trust globally — cross-tenant DB access

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-providers/src/externalsvc/postgres_cluster.rs:217-256, node_command()`
- **What's wrong:** Each data node's startup script appends `hostssl/host replication pgautofailover_replicator 0.0.0.0/0 trust` (lines 218-225) AND a catch-all `hostssl/host all all 0.0.0.0/0 md5` (lines 252-255). The replication `trust` rules let anyone on the network stream WAL as `pgautofailover_replicator` with NO password (full data exfiltration / base-backup). The `all all md5` rule exposes every role — including the auto-created LOGIN SUPERUSER app role — to password auth from the entire internet, and the data port is published on `0.0.0.0` (agent service_handlers.rs:110-123). The comment even acknowledges it is 'from anywhere on the network the cluster already trusts' — but the rule is literally `0.0.0.0/0`.
- **Scenario:** An attacker who can reach a cluster data port (overlay neighbour, adjacent worker, or a misconfigured/public worker IP) either streams the entire database as pgautofailover_replicator with no password, or brute-forces/uses a leaked superuser password against the md5 catch-all from anywhere, gaining full read/write on another tenant's managed database.
- **Recommendation:** Scope every appended hba rule to the cluster overlay subnet instead of `0.0.0.0/0`. Remove the replication `trust` lines in favour of cert/scram auth (pg_auto_failover issues replication certs). Use `scram-sha-256` not `md5`. Bind data ports to the overlay interface and firewall the worker. Generate per-role pg_hba entries instead of an `all all` catch-all.

### `cluster-3` — 🟠 HIGH — Control-plane→cluster TLS accepts any certificate and falls back to plaintext — data-plane MITM

- **Verification:** ✅ verified inline by analyst (verifier rate-limited)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-query-postgres/src/lib.rs:23-63 (AcceptAllVerifier) and :90-99 (plaintext fallback)`
- **What's wrong:** `connect_with_self_signed_tls` (used by the role reconciler postgres_role_reconciler.rs:441, probe_cluster services.rs:2247, and cluster_health) installs `AcceptAllVerifier`, whose `verify_server_cert`/`verify_tls12/13_signature` all return success unconditionally — no hostname, CA, or pin check. `PostgresSource::connect` additionally logs 'TLS connection failed, falling back to plain connection' and connects in cleartext. Because the monitor uses trust auth (cluster-1), TLS is the ONLY thing standing between an attacker and autoctl_node; accepting any cert removes it.
- **Scenario:** An on-path attacker on the underlay/WireGuard segment presents a self-signed cert; the control plane accepts it, and the attacker proxies/inspects/alters the monitor query stream (reported_state of nodes) — feeding the reconciler false primary/replica states so it points the `primary.<svc>.temps.local` VIP at the attacker's host, redirecting all application writes (data-plane MITM). The plaintext fallback makes passive interception of any password-auth member trivial.
- **Recommendation:** Pin the cluster's self-signed CA: have the monitor/nodes use a per-cluster CA whose cert temps stores at create time, and verify against it (rustls with a custom root store) instead of AcceptAllVerifier. Remove the silent plaintext fallback for cluster connections, or gate it behind an explicit, logged, non-default flag.

### `cluster-6` — 🟠 HIGH — All cluster members run the app role as LOGIN SUPERUSER, reachable via the global md5 hba rule

- **Verification:** ✅ verified inline by analyst (verifier rate-limited)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-providers/src/externalsvc/postgres_cluster.rs:290-293 (CREATE/ALTER ROLE ... LOGIN SUPERUSER) combined with :252-255 (host all all 0.0.0.0/0 md5)`
- **What's wrong:** The node startup script creates/alters the configured application user as `LOGIN SUPERUSER` ('CREATE ROLE "${POSTGRES_USER}" LOGIN SUPERUSER ...' / 'ALTER ROLE ... WITH LOGIN SUPERUSER'). That same role is exposed to password auth from `0.0.0.0/0` by the catch-all md5 rule. A 16-char alphanumeric auto-generated password (postgres.rs:172-179, ~95 bits) is the only barrier, and the connection string embeds it in plaintext env/URL. Granting the tenant-facing role SUPERUSER means a single credential compromise yields OS-level reach (COPY PROGRAM, untrusted PL, etc.) on the database host, not just data access.
- **Scenario:** If the app connection string leaks (env var in a deploy log, SSRF, a captured plaintext fallback connection from cluster-3) the attacker gets Postgres SUPERUSER on a network-reachable port, enabling `COPY ... TO PROGRAM` for arbitrary command execution inside the cluster container and lateral movement.
- **Recommendation:** Do not provision the tenant app role as SUPERUSER — grant only the privileges the app needs (ownership of its database, CREATE on its schema). Keep SUPERUSER for an internal break-glass role that is NOT exposed by the catch-all hba. Combine with cluster-2's subnet-scoping so the role is only reachable from the overlay.

### `cluster-4` — 🟡 MEDIUM — Monitor-unreachable 'is live primary' guard is defeated by static role model — primary can be removed/promoted-over

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-providers/src/services.rs:2094-2102 (member_is_live_primary) vs 4305-4309 / 5091-5095 (role always stored as 'replica')`
- **What's wrong:** When the monitor is unreachable, `member_is_live_primary` falls back to `is_role_primary(&member.role)` to refuse deleting the last-known primary. But by design every data member is persisted with `role = 'replica'` (initialize_cluster and add_cluster_member both store 'replica'; 'primary' is runtime-only via live_state). So the fallback returns `false` for every data node when the monitor is down — exactly the partition scenario where the guard matters. `remove_cluster_member` (services.rs:5749) then permits removing the actual writable primary, and the quorum check at 5763-5781 counts `service_members` rows (including dead/offline members), so it does not compensate.
- **Scenario:** During a network partition or monitor outage, an operator (or automation) removes what is in fact the live primary because its stored role is 'replica'; pg_auto_failover loses its writable node and the cluster cannot elect a new primary cleanly, causing write outage / potential data loss on reconvergence.
- **Recommendation:** Persist the last-known-primary identity (e.g. a `last_known_primary_member_id` on the service, refreshed by the reconciler) and use THAT in the monitor-down fallback instead of the now-always-'replica' role column. Make the quorum check count only members in a healthy/running live_state, and block destructive membership changes entirely while the monitor is unreachable unless an explicit force flag is passed.

### `cluster-5` — 🟡 MEDIUM — Reconciler picks 'first primary by nodeid' on split-brain instead of fencing — VIP can point at a stale primary

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** reliability · **Confidence:** medium · **Location:** `crates/temps-providers/src/externalsvc/postgres_role_reconciler.rs:140-157 (drafts_for_snapshot) and 622-638 (test documents the behaviour)`
- **What's wrong:** When the monitor briefly reports two nodes in a primary state (documented mid-failover edge case), the reconciler does not treat it as an error or fence either node — it deterministically picks `min_by_key(nodeid)` and publishes that as `primary.<svc>.temps.local` (TTL 5s). The comment says 'apps will just retry against whichever still accepts target_session_attrs=read-write', but there is no verification that the chosen node is actually the writable one; it could pick a demoted/old primary whose nodeid is lower. There is no explicit fencing of the old primary at the temps layer — it relies entirely on pg_auto_failover's own demotion.
- **Scenario:** During a flapping failover the lower-nodeid node is the OLD primary that hasn't yet been demoted; the reconciler points the write VIP at it for up to a tick. Apps using the single-VIP connection string (cluster_connection_string all_fqdn branch) can briefly write to a node that is about to be demoted, risking lost writes / divergence on reconvergence.
- **Recommendation:** On observing >1 primary, prefer the node the monitor marks as the current `wait_primary`→`primary` transition winner, cross-check with `target_session_attrs=read-write` reachability before publishing, and if ambiguous, withhold the primary record (let libpq's multi-A target_session_attrs select the writable one) rather than guessing by nodeid. Log split-brain observations at WARN and surface them in cluster-health.

### `cluster-7` — 🟡 MEDIUM — Best-effort 'drop node' on remove can orphan a member in the monitor formation, blocking later failovers

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-providers/src/services.rs:5798-5821 (drop_node_from_monitor best-effort) and 5783-5788 (ordering comment)`
- **What's wrong:** remove_cluster_member calls `drop_node_from_monitor` first, but on failure it only logs a warning ('cluster may need manual `pg_autoctl drop node` after cleanup') and proceeds to delete the container and the `service_members` row anyway. The code itself documents that a node left in the monitor 'still expect[s] it to participate in quorum (e.g. report_lsn during a later failover), wedging the FSM.' So a transient monitor hiccup during removal permanently orphans a ghost node in pg_auto_failover while temps forgets it ever existed (row deleted), and there is no reconciliation that re-attempts the drop.
- **Scenario:** A replica is removed while the monitor is briefly unreachable; the drop fails, the row+container are deleted, and the ghost node remains registered in pgautofailover.node. On the next real primary failure, the monitor waits on report_lsn from the now-deleted node, stalling promotion and causing a write outage the operator cannot fix from the UI (the member no longer exists in temps).
- **Recommendation:** Do not delete the service_members row when the monitor drop fails — keep it in a `pending_drop` state and have a background reconciler retry `pg_autoctl drop node` until the monitor confirms removal, or refuse the delete (return 503) when the monitor is unreachable so the FSM never gets a ghost. Surface orphaned monitor nodes in cluster-health.

### `cluster-8` — ⚪ LOW — Monitor identity / membership is not bound to a node ownership check — rogue node_id in member requests trusted as-is

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** security · **Confidence:** medium · **Location:** `crates/temps-providers/src/services.rs:4184-4205 (member_specs from request node_id) and 5332-5342 (add member node_id), handlers/handlers.rs:466-479`
- **What's wrong:** Cluster create and add-member accept an arbitrary `node_id` from the (authenticated, permissioned) request and resolve it straight to `nodes.private_address` to place a container, without verifying the node is healthy, joined, or otherwise eligible for this cluster — the only check is that the node row exists (`find_by_id(...).ok_or(... Node not found)`). There is no validation that the target node is `online`/approved, so a member can be scheduled onto a stale or partially-decommissioned node, and the monitor will then advertise that node's underlay IP as a cluster endpoint via the reconciler's `nodes.private_address` join (postgres_role_reconciler.rs:281-287).
- **Scenario:** An operator (or compromised admin token) targets a node_id that has been re-I'd or is mid-decommission; the cluster member lands on the wrong host, and the role reconciler publishes that host's private_address as the primary/replica A record, sending application traffic to an unintended machine. Not cross-tenant, but a placement/integrity weakness.
- **Recommendation:** Validate the target node's status (joined + online/approved) before scheduling a cluster member onto it, and re-validate node identity/address freshness in the reconciler before publishing its private_address as a DNS target. Reject node_ids that are not in an eligible state with a 400.

**✅ Strengths already in place (do NOT regress):**

- Every cluster HTTP handler enforces authentication + authorization: permission_guard!(auth, ExternalServices{Read,Write,Create,Delete}) plus super::metrics_handlers::assert_service_owned_by_caller(id, ...) on get/update/delete/start/retry/add/remove/promote, and fleet-wide list endpoints additionally call deny_deployment_token! (handlers/handlers.rs:384-388, 1102-1105).
- All write operations on membership (add/remove/promote/retry) emit audit logs with user_id, ip, user_agent (ExternalServiceClusterMember*Audit), so membership changes are traceable.
- The worker agent API is bearer-token authenticated with a SHA-256 hashed token and a constant-time comparison that avoids timing side channels, and length-mismatch short-circuits safely (temps-agent/src/auth.rs:25-49, require_agent_auth).
- Ordinal allocation is race-safe: a unique (service_id, ordinal) constraint (m20260428_000001) backs a bounded 8-attempt retry loop that recomputes the next free ordinal on collision instead of blindly using MAX+1 (services.rs:5078-5151).
- remove_cluster_member enforces real safety invariants: refuses to remove the monitor, refuses to remove the live primary (member_is_live_primary), and enforces a 2-data-member quorum floor before allowing removal (services.rs:5736-5781); it also drops the node from the monitor before deleting the container to avoid FSM wedging.
- promote_cluster_member validates topology, engine, ownership (member.service_id == service_id), rejects promoting the monitor or the already-primary node, and requires the target to be running before issuing pg_autoctl perform promotion (services.rs:5946-6012).
- The role reconciler writes role/VIP DNS records as a single atomic replace_endpoints_for_owner batch, so resolvers never observe a 'primary removed but new primary not yet inserted' transient (postgres_role_reconciler.rs:18-23, 336-338), and uses a short 5s primary TTL so failover propagates quickly.
- Postgres password handling for cluster startup is hardened against SQL/shell injection: is_valid_pg_password rejects ', \, $, null, newline and control chars, and the app-role SQL is written to a tempfile rather than inlined (parameter_strategies.rs:54-68, postgres_cluster.rs:286-309).
- cluster_connection_string correctly excludes the monitor from the app connection string and url-encodes the password (postgres_cluster.rs:550-560, 810-811 test).

## Node enrollment & identity

> I traced the full worker-node enrollment and identity path: CLI join (crates/temps-cli/src/commands/join.rs) → control-plane register endpoint (crates/temps-deployments/src/handlers/nodes.rs::register_node, mounted public at /api/internal/nodes/register in crates/temps-cli/src/commands/serve/console.rs) → NodeService::register (crates/temps-deployments/src/services/node_service.rs) → nodes table (entity crates/temps-entities/src/nodes.rs, migration m20260305_000001). I also reviewed join-token generation/revocation (crates/temps-config/src/handler.rs), the agent-side bearer auth (crates/temps-agent/src/auth.rs, server.rs), and the route-sync / S3-credential / edge-routes endpoints that reuse the node token.  Overall posture: enrollment is gated by a SINGLE cluster-wide shared join token (SHA-256 hashed, compared in constant time) and registration is "one-shot trust" — once a node is in the table, it authenticates forever with a long-lived bearer token that it (or the relay) chose, with no rotation, expiry, or revocation, and no per-node identity verification. The biggest weaknesses are: (1) any holder of the shared join token can overwrite an existing node's identity by name (token/address/wg-key takeover), (2) the join token never expires and is not single-use, (3) registration is unauthenticated beyond the shared token, unthrottled, and on a publicly-exposed endpoint, (4) the data/control channel to direct-mode agents is plain HTTP carrying the bearer token, and (5) no audit logging of enrollment/reconnection/removal. Crypto primitives, SSRF address validation, and config-file permissions are done well.

### `enroll-1` — 🟠 HIGH — Node re-registration by name lets a join-token holder hijack an existing node's identity

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-deployments/src/services/node_service.rs:78-108`
- **What's wrong:** register() resolves an existing node purely by name and, on match, overwrites its security-critical fields with caller-supplied values: `let existing = nodes::Entity::find().filter(nodes::Column::Name.eq(&request.name)).one(...).await?; if let Some(existing_node) = existing { let mut active = existing_node.into(); active.token_hash = Set(request.token_hash); active.token_encrypted = Set(...); active.address = Set(request.address); active.private_address = Set(request.private_address); active.wg_public_key = Set(request.wg_public_key); active.status = Set("active"); ... }`. The only gate upstream (register_node, nodes.rs:585-619) is the shared cluster join token. There is no check that the re-registering caller is the same machine/identity that originally enrolled that node — no proof-of-possession of the prior token, no WG-key continuity check.
- **Scenario:** An attacker who obtains the cluster join token (it is long-lived, shared, and passed on CLI / env / relay — see enroll-2/enroll-3) POSTs /api/internal/nodes/register with name="worker-1" (an existing node), their own agent token, and address=attacker-host. The control plane silently rebinds node worker-1 to the attacker's address/token and marks it active. The control plane then dials the attacker's `address` to deploy containers, exec commands, and hands it S3 credentials (get_s3_credentials, nodes.rs:897) and route tables — full impersonation of a legitimate worker and redirection of deployments/secrets to attacker-controlled infrastructure.
- **Recommendation:** Treat re-registration as a privileged operation: require the existing node to prove possession of its current token (or WG private key) before any field is overwritten, or bind node identity to an immutable first-seen credential (e.g. the WireGuard public key / a per-node enrollment cert) and reject a register that changes token_hash/wg_public_key/address for an existing name unless authenticated as that node. At minimum, gate identity-changing re-registration behind admin approval and audit-log it.

### `enroll-2` — 🟠 HIGH — Cluster join token never expires, is not single-use, and is shared across all nodes

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-config/src/handler.rs:803-850; crates/temps-core/src/app_settings.rs:413-421`
- **What's wrong:** generate_join_token() creates one 32-byte token, stores only `s.multi_node.join_token_hash = Some(token_hash)` and returns the plaintext. MultiNodeSettings has exactly two fields — `join_token_hash: Option<String>` and `private_address` — with no expiry, issued-at, max-uses, or per-node binding (app_settings.rs:413-421). The same secret enrolls every node forever until an admin manually regenerates (which simultaneously invalidates ALL nodes' ability to re-enroll, since there is only one). register_node compares the provided token's hash to this single stored hash (nodes.rs:585-619).
- **Scenario:** The join token leaks once (CI logs, shell history, a compromised worker's process args, the relay) and remains valid indefinitely for enrolling unlimited rogue nodes or hijacking existing ones (enroll-1). Because it is cluster-wide and non-expiring, the operator cannot scope or time-box a single node's enrollment, and revoking it to contain a leak breaks legitimate re-enrollment for the whole fleet.
- **Recommendation:** Issue short-lived, single-use (or N-use) enrollment tokens with an explicit expiry and optional name/label binding, tracked in a dedicated table (consumed_at / expires_at), separate from the steady-state node token. Keep the per-node bearer token distinct from the enrollment token so revoking enrollment does not invalidate running nodes.

### `enroll-4` — 🟠 HIGH — Direct-mode agent channel is plaintext HTTP carrying the long-lived bearer token

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/join.rs:121-141; crates/temps-agent/src/server.rs:367-386; crates/temps-deployer/src/remote.rs:43-46`
- **What's wrong:** In direct mode the CLI advertises the agent at `http://{private_address}:{port}` (join.rs:138) and the generated agent token (generate_token, join.rs:121/369) becomes the node's permanent credential. The agent server binds a plain TCP listener and `axum::serve(listener, router)` with no TLS (server.rs:367-386). The control plane's RemoteDeployer dials that address and additionally sets `.danger_accept_invalid_certs(temps_core::tls::insecure_tls_enabled())` (remote.rs:43-46), so even if HTTPS were used it can be downgraded to accept invalid certs cluster-wide.
- **Scenario:** Direct mode is documented as traversing user-managed networking that 'may traverse the public internet' (join.rs comment). The agent's bearer token and every deploy/exec/S3-credential request are sent in cleartext HTTP. Any on-path attacker between control plane and worker captures the agent bearer token (which authenticates ALL agent endpoints — deploy container, exec, file ops) and can then drive the worker directly, or MITM the deploy traffic to inject a malicious container image/command. The insecure_tls toggle compounds this by disabling cert validation fleet-wide.
- **Recommendation:** Require an encrypted, mutually-authenticated channel for the agent in direct mode (agent-side TLS with a cert pinned at enrollment, or mandate WireGuard). Do not advertise http:// agent URLs; reject non-TLS agent addresses unless the operator explicitly opts in per-node. Scope insecure_tls to a single node rather than a global flag, and warn loudly.

### `enroll-3` — 🟡 MEDIUM — Registration endpoint is publicly exposed and unthrottled (token brute-force / enrollment DoS)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/serve/console.rs:1826-1828; crates/temps-deployments/src/handlers/nodes.rs:278-291`
- **What's wrong:** node_routes (including /internal/nodes/register) are merged into `public_router` with the explicit comment "Agent-facing node + route-sync routes are public (workers anywhere on the internet POST to them with bearer tokens)" (console.rs:1826-1828). configure_routes() applies no rate-limiting or IP allow-listing layer (nodes.rs:278-291); the only registration control is the single shared-token equality check. No per-IP throttle, no lockout, no cap on node count.
- **Scenario:** An unauthenticated internet attacker can hammer /api/internal/nodes/register to (a) attempt to guess/confirm the join token offline-style at line speed (no backoff), and (b) flood the nodes table / overlay-CIDR allocator (allocate_overlay_cidr, nodes.rs:714) and telemetry with bogus registrations, exhausting the compute CIDR pool and node-name space, degrading scheduling. The endpoint also does heavy work (settings read, encryption, CIDR allocation) per request with no rate limit.
- **Recommendation:** Put the enrollment endpoint behind rate-limiting and, where possible, network restriction (relay/WG-only or an admin allow-list). Add per-IP throttling + exponential backoff on repeated invalid-join-token responses, and cap concurrent pending registrations.

### `enroll-5` — 🟡 MEDIUM — No node-token rotation or post-enrollment re-authentication — enrollment is permanent one-shot trust

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-deployments/src/handlers/nodes.rs:799-821,897-916; crates/temps-routes/src/route_sync.rs:257-288; crates/temps-agent/src/auth.rs:19-29`
- **What's wrong:** After enrollment the node proves identity only by replaying the same static bearer token whose SHA-256 is stored in nodes.token_hash: heartbeat (nodes.rs:815-816), S3 credentials (nodes.rs:910-911), route-sync (route_sync.rs:279-280), and the agent side (auth.rs verify). There is no rotation mechanism, no token TTL, and no re-attestation. The same plaintext is also stored encrypted-at-rest (token_encrypted) so the CP can call the agent — so the credential is symmetric and reused in both directions indefinitely.
- **Scenario:** A node token captured at any point (cleartext channel per enroll-4, agent process args, a backup of the nodes table, or the relay) authenticates the holder as that node forever. There is no way to rotate a single node's token without a full re-enroll-by-name (which itself is the hijack vector of enroll-1), and the control plane cannot force expiry. A long-lived stolen token = persistent worker impersonation and ongoing receipt of S3 credentials/route tables.
- **Recommendation:** Add per-node token rotation (e.g. CP issues a new token on a schedule / on demand, agent rotates and re-stores agent.json) with an expiry column, and support immediate revocation that the heartbeat/auth path honors. Prefer asymmetric node identity (agent holds a private key, CP verifies a signature) so the long-lived secret is never transmitted.

### `enroll-6` — 🟡 MEDIUM — Removing a node does not revoke its credentials or WireGuard peer access

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** medium · **Location:** `crates/temps-deployments/src/handlers/nodes.rs:1720 (admin_remove_node); crates/temps-deployments/src/services/node_service.rs:84-108`
- **What's wrong:** admin_remove_node soft-deletes / removes the node row, but there is no revocation list and re-registration is name-keyed and resurrects/overwrites by name (node_service.rs:84-108). Auth everywhere is 'does token_hash match a row with this node_id/name'. Once a node is removed, its previously-issued WireGuard peer config, agent token, and the still-valid shared join token are not invalidated; nothing prevents the same machine (or an attacker who kept the token) from simply re-registering and being trusted again.
- **Scenario:** An operator decommissions a compromised or departing worker via the admin UI, believing it is now locked out. The node's agent token and join token remain valid, so the machine re-POSTs /internal/nodes/register (same name) and is immediately re-trusted as active, regaining deploy targeting and S3-credential access. Removal provides no real containment.
- **Recommendation:** On node removal, explicitly revoke the node's token (mark token_hash dead / add to a denylist), tear down its WireGuard peer, and require a fresh single-use enrollment token (enroll-2) for any re-join. Block name reuse from silently resurrecting a removed node.

### `enroll-7` — 🟡 MEDIUM — Enrollment, reconnection, and node removal are not audit-logged on the control plane

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-deployments/src/handlers/nodes.rs:573-725 (register_node), 1720 (admin_remove_node)`
- **What's wrong:** grep for 'audit'/'Audit' across the entire node handlers file returns zero matches. register_node emits only anonymous product telemetry (nodes.rs:693-698, role label only) and an info! log line; there is no AuditService entry recording who enrolled which node, from which IP, with what address/key, nor for identity-changing reconnections or removals. This contradicts the project rule 'Add audit logging for all write operations (CREATE, UPDATE, DELETE)'.
- **Scenario:** After a rogue-enrollment or node-hijack (enroll-1), responders have no tamper-evident audit trail of when a node's address/token/wg-key changed or which source IP performed the registration. Detection and forensics rely on best-effort tracing logs, which are not the audit log and may be rotated. Silent identity rebind is effectively invisible.
- **Recommendation:** Add audit-log entries (source IP, node name/id, address, wg_public_key fingerprint, new-vs-reconnect, join-token-id once tokenized) for register_node, identity-changing reconnects, and admin_remove_node. Surface reconnections that change token_hash/address/wg key as high-signal audit events.

### `enroll-8` — 🟡 MEDIUM — Relay-mode trust: control plane accepts a node token and identity chosen by the third-party relay/client without binding

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** medium · **Location:** `crates/temps-cli/src/commands/join.rs:230-293`
- **What's wrong:** In relay mode the agent_token, assigned_ip, and node_id all come from the relay's RelayJoinResponse (join.rs:231-243), and the CLI then registers with the control plane sending that relay-supplied agent_token plus a client-chosen node_name and the relay-assigned private_address (join.rs:284-293). The control-plane register_node validates only the shared join token; it does not verify that the registration genuinely originated from the relay for this cluster, nor bind the node's WG public key (sent in the body) to the assigned IP at a trust anchor. The node_name and private_address in the register body are attacker-controllable independent of what the relay assigned.
- **Scenario:** A client with the join token (or a malicious/compromised relay) can register with an arbitrary node_name (enabling the enroll-1 hijack), an arbitrary private_address, and a wg_public_key of their choosing — the control plane will tunnel and route to it. Because the CP trusts the relay/client-asserted fields rather than cryptographically binding identity (WG key) to the assigned overlay address, a node can claim an IP/name it was not assigned.
- **Recommendation:** Have the control plane (not just the relay) be the authority that binds wg_public_key ↔ assigned private_address ↔ node identity, and reject register bodies whose private_address/name don't match a CP-side allocation tied to a single-use enrollment token. Authenticate the relay path so the CP can distinguish a genuine relay-mediated join from a raw token-holder POST.

**✅ Strengths already in place (do NOT regress):**

- Join token is verified with a constant-time hash comparison: register_node uses constant_time_eq(provided_hash, stored_hash) (nodes.rs:591), and route-sync/agent paths use subtle::ConstantTimeEq (route_sync.rs:280, agent/auth.rs), preventing timing oracles. The local constant_time_eq operates on fixed-length 64-char SHA-256 hex, so its length early-return is not a leak.
- Join token is never stored in plaintext — only its SHA-256 hash is persisted (app_settings.rs:414 comment 'never store plaintext'), and the masked settings response elides it (config/handler.rs:211-214, has_join_token boolean only). The plaintext is shown exactly once at generation.
- Registration is closed by default: if no join token hash is configured, ALL node registrations are rejected (nodes.rs:612-618) rather than open-enrollment.
- Strong CSPRNG for tokens: both the join token (config/handler.rs:811-815) and the agent token (join.rs:369-374) use rand 32 random bytes hex-encoded.
- Solid SSRF hardening on enrollment input: validate_node_private_address and is_safe_api_address (nodes.rs:496+) reject loopback, link-local, broadcast, unspecified, and cloud-metadata IPs for both private_address and the agent address before persistence (nodes.rs:626-656).
- Agent config file written with restrictive permissions — dir 0700 and agent.json 0600 (join.rs:86-102), protecting the stored bearer token at rest on the worker.
- CLI deliberately refuses to honor the server-side insecure_tls opt-in for the join/register requests, always requiring valid TLS for the join-token-carrying call (join.rs:127-130, 264-270 comments), reducing MITM of the enrollment exchange itself.
- Per-node agent token is encrypted at rest in token_encrypted via AES-256-GCM EncryptionService before storage (nodes.rs:660-668), not stored in plaintext alongside the hash.

## WireGuard mesh cryptography & key handling

> I audited the WireGuard mesh crypto and key lifecycle across crates/temps-wireguard/src/lib.rs (key generation, interface/peer programming), crates/temps-cli/src/commands/join.rs (relay + direct join, WG bring-up), crates/temps-agent/src/lib.rs (AgentConfig persistence), crates/temps-network/src/{config.rs,allocator.rs} (compute-CIDR allocation/peer list), and the control-plane node lifecycle in crates/temps-deployments/src/{handlers/nodes.rs,services/node_service.rs}.  Key generation is sound: x25519_dalek::StaticSecret::random_from_rng(OsRng) is a proper CSPRNG (lib.rs:126), keys are 32-byte Curve25519, and the public key is derived correctly. On-disk handling of the agent config is good (0700 dir / 0600 file in join.rs:80-105). However the audit surfaced a structural gap: WireGuardManager is ONLY ever invoked from the worker side in join.rs — the control plane (`temps serve`) never programs a WireGuard interface or adds the worker as a peer (grep shows no init_interface/add_peer callers outside join.rs and tests). The worker trusts the relay-supplied control-plane public key blindly (no pinning/out-of-band verification), there is no preshared key, no key rotation/revocation on node removal, the WG private key is never persisted so the tunnel cannot survive a restart, and the join token is a single static cluster-wide shared secret. Net posture: the data-plane crypto primitive itself is correct, but the mesh trust model, key lifecycle, and crypto/control-plane coupling are weak.

### `wg-1` — 🟠 HIGH — Relay-supplied control-plane public key is trusted blindly (mesh MITM / rogue control plane)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/join.rs:231-257`
- **What's wrong:** In relay mode the worker takes the control plane's WireGuard identity entirely from the relay's JSON response and adds it as a peer with no authentication: `control_plane_wg_pubkey: String` (line 233) is read from `RelayJoinResponse` and fed directly into `WireGuardPeer { public_key: relay_response.control_plane_wg_pubkey, endpoint: relay_response.control_plane_endpoint, ... }` then `wg_manager.add_peer(&peer).await?` (lines 252-257). There is no pinning of the control-plane pubkey to the join token, no admin-visible fingerprint, and no signature. The relay at api.temps.sh (a third party in Temps Cloud) is fully trusted to hand out the cryptographic peer identity.
- **Scenario:** A malicious or compromised relay (or anyone who can MITM the worker's HTTPS call to api.temps.sh and substitute a response) returns its own WireGuard pubkey and endpoint as `control_plane_wg_pubkey`/`control_plane_endpoint`. The worker establishes an encrypted tunnel to the attacker instead of the real control plane. The attacker now sits as the control plane for that worker — it can drive deploy/exec/route-sync calls to the agent (taking over the Docker host) and harvest the agent token the worker subsequently registers with.
- **Recommendation:** Bind the control-plane WireGuard identity to a secret the relay does not control: have the cluster admin embed (or fingerprint) the control-plane WG public key in the join material, and have the worker verify the relay-returned pubkey against that pinned value before calling add_peer. At minimum, print the control-plane pubkey fingerprint for out-of-band confirmation, and prefer Direct mode (which already mandates TLS) for high-trust deployments.

### `wg-3` — 🟠 HIGH — Removing a node never tears down its WireGuard peer (no crypto revocation)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-deployments/src/services/node_service.rs:336-348`
- **What's wrong:** `NodeService::remove` only deletes the database row: `nodes::Entity::delete_by_id(node_id).exec(...).await?` (lines 337-339) followed by a log line. There is no call to `WireGuardManager::remove_peer` (grep confirms remove_peer has zero non-test callers). The same is true for `mark_offline`/`mark_draining` (node_service.rs:220,236) — they only flip a status string. A node's WireGuard public key stays an authorized peer on any interface that ever added it.
- **Scenario:** An operator removes a compromised or decommissioned worker from the cluster (DB row deleted). That node's WireGuard keypair is still cryptographically authorized: it can still complete a handshake and send/receive inside the overlay (subject to allowed_ips) until interfaces are manually torn down or hosts rebooted. Revocation is purely a scheduling/DB concept, not a crypto one, so a stolen node key remains valid indefinitely.
- **Recommendation:** On node removal/decommission, drive `remove_peer(wg_public_key)` on every interface that holds it (today that means at least the worker-side teardown, and the control-plane interface once wg-6 is fixed), and re-push the peer list to surviving nodes so the revoked key is dropped mesh-wide. Treat key revocation as a first-class step of node removal.

### `wg-4` — 🟠 HIGH — WireGuard private key is never persisted — tunnel cannot survive an agent restart/reboot

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/join.rs:204-249, crates/temps-agent/src/lib.rs:71-92`
- **What's wrong:** `temps join` generates the keypair (`keypair = wg_manager.generate_keypair()`, join.rs:204), brings up a userspace (boringtun) interface with it (`init_interface(our_ip, &keypair.private_key)`, join.rs:248), then SAVES the agent config and exits — the module doc says 'saves the agent config ... and exits' (join.rs:7-8). The persisted `AgentConfig` (temps-agent/src/lib.rs:71-92) contains listen_address, token, node_name, control_plane_url, node_id, labels, dns_data_dir — but NO WireGuard private/public key. `temps agent` (agent.rs) never calls init_interface. Because the WG interface lives in the short-lived `temps join` process's userspace, it dies when that process exits, and the key needed to recreate it is gone.
- **Scenario:** After `temps join` exits (or the host reboots, or the join process is killed), the userspace WireGuard tunnel is gone and there is no stored private key to re-establish it. The worker drops off the relay-mode overlay and cannot reach the control plane over WireGuard until the operator re-runs `temps join` with a fresh key (which also generates a brand-new identity, compounding wg-3's stale-peer problem). The cluster silently loses the node's data-plane connectivity with no recovery path.
- **Recommendation:** Persist the WireGuard private key (encrypted at rest, like node tokens are via EncryptionService) and assigned IP in the agent config or a 0600 key file, and have `temps agent` re-create/verify the WireGuard interface from that persisted key on every startup so the tunnel is durable across restarts and reboots.

### `wg-2` — 🟡 MEDIUM — No preshared key (PSK) on any WireGuard peer

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-wireguard/src/lib.rs:220-241`
- **What's wrong:** Peers are constructed with `let mut wg_peer = defguard_wireguard_rs::peer::Peer::new(key);` and only `set_endpoint`, `allowed_ips`, and `persistent_keepalive_interval` are set (lines 220-241). The `preshared_key` field is never populated, and `WireGuardPeer` (lib.rs:37-45) has no PSK field at all. WireGuard's optional 256-bit symmetric PSK (the only post-quantum / defense-in-depth layer in the protocol) is therefore unused on every tunnel in the mesh.
- **Scenario:** If an attacker records mesh traffic today and a future break of Curve25519 (or a private-key compromise) occurs, there is no second symmetric secret protecting the session — the entire mesh's confidentiality reduces to X25519 alone. A PSK would also harden against the wg-1 blind-trust issue by requiring a shared secret the relay never sees.
- **Recommendation:** Add an optional per-cluster or per-peer preshared key, derived from or exchanged alongside the join token (which the relay-MITM cannot forge), and set `wg_peer.preshared_key` on both ends. This provides defense-in-depth and partially mitigates the relay trust gap.

### `wg-5` — 🟡 MEDIUM — Join token is a single static cluster-wide shared secret (no per-node, single-use, or expiring tokens)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-deployments/src/handlers/nodes.rs:585-619, crates/temps-core/src/app_settings.rs:415`
- **What's wrong:** Registration validates against one global `settings.multi_node.join_token_hash` (app_settings.rs:415; checked in nodes.rs:585-599 via constant_time_eq over the hex hash). The same plaintext join token is shared by every worker, has no expiry, and is not consumed on use. It is also what gates relay join (join.rs:215 `join_token`) and gets sent on every register. Anyone who obtains it can enroll arbitrary workers indefinitely.
- **Scenario:** The join token leaks (CI logs, a decommissioned node's disk, shell history, an over-broad env var). An attacker registers a rogue worker node into the cluster at any later time; once active it receives deployments/secrets and participates in the overlay. There is no way to invalidate just the leaked token without rotating it for the entire fleet, and no audit of which token enrolled which node.
- **Recommendation:** Move to per-node, single-use, time-bounded join tokens (store issued tokens with status/expiry and mark consumed on first successful register), and support rotation/revocation of an individual token. Keep the constant-time compare, which is already correct.

### `wg-6` — 🟡 MEDIUM — Control plane never programs a WireGuard interface — mesh peering is one-directional / asymmetric

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-wireguard/src/lib.rs:142-259 (no control-plane caller), crates/temps-deployments/src/handlers/nodes.rs:670-687`
- **What's wrong:** `init_interface`/`add_peer` have no callers anywhere except the worker-side join.rs and tests (verified by grep across crates/). The control plane's register handler stores the worker's `wg_public_key` into the DB (`wg_public_key: request.wg_public_key`, nodes.rs:677; persisted in node_service.rs:92,117) but never feeds it into a WireGuardManager to add the worker as a peer or to stand up its own wg0. So in relay mode the worker adds the control plane as a peer, but the control plane never reciprocally adds the worker. The stored wg_public_key is effectively dead data on the control-plane side, and there is no code path that builds the symmetric peering required for a working WireGuard tunnel.
- **Scenario:** Either (a) the actual control-plane WireGuard configuration happens entirely out-of-band/undocumented, meaning peer add/remove (and thus revocation, wg-3) is never automated and stale or rogue peers accumulate; or (b) relay-mode tunnels never come up symmetrically at all. Both are reliability/security problems: there is no programmatic source of truth that adds the right worker key and removes a revoked one on the control-plane side.
- **Recommendation:** Make the control plane own a WireGuardManager: on node register, add the worker (verified wg_public_key, allowed_ips = its /32) as a peer; on node remove, drop it (see wg-3). Drive the peer set from the DB so the mesh membership is reconciled and auditable rather than implicit.

### `wg-7` — ⚪ LOW — next_available_ip ignores subnet mask — only varies the last octet, misallocating for masks other than /24

- **Verification:** ⚠️ disputed (1/2 — needs human call)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-wireguard/src/lib.rs:321-340`
- **What's wrong:** `next_available_ip` builds candidates as `format!("{}.{}", self.subnet_prefix, last_octet)` for `last_octet in 1..255` (lines 322-323), where `subnet_prefix` is hard-coded to the first three octets (`format!("{}.{}.{}", ...)`, lib.rs:95). It never consults `self.subnet_mask`. For any configured WG subnet that is not a /24 (e.g. a /16 set via TEMPS_WG_SUBNET, or a /25), it can only ever hand out 254 addresses inside a single /24 and the SubnetExhausted error (lines 337-339) reports the configured mask, masking the real cap.
- **Scenario:** An operator sets TEMPS_WG_SUBNET=10.100.0.0/16 expecting ~65k mesh IPs. The allocator caps the cluster at 254 nodes and then errors as 'subnet exhausted /16', a confusing reliability ceiling and a misleading error. Conversely a /25 subnet would let it hand out addresses outside the intended range.
- **Recommendation:** Compute candidate addresses from the full network/mask (e.g. iterate the host range of the parsed CIDR via ipnet, which the network crate already depends on) instead of only mutating the last octet; honor subnet_mask in both allocation and the exhaustion message.

**✅ Strengths already in place (do NOT regress):**

- Private key generation uses a proper CSPRNG: x25519_dalek::StaticSecret::random_from_rng(rand::rngs::OsRng) with x25519-dalek 2.0 (lib.rs:126-127). StaticSecret handles Curve25519 clamping internally, and a unit test verifies keys are 32 bytes, base64-valid, unique, and that the public key correctly derives from the private key (lib.rs:417-447).
- Agent config is written with tight filesystem permissions: 0700 on ~/.temps and 0600 on agent.json (join.rs:86-102), so secrets at rest are owner-only on Unix.
- Worker WireGuard private keys never leave the host — only the public key is sent to the relay/control plane (join.rs:218,291). The private key is used locally to configure the interface and is not transmitted.
- allowed_ips for the control-plane peer is correctly scoped to a single /32 (`format!("{}/32", relay_response.control_plane_ip)`, join.rs:255), preventing that peer from claiming a broader range of overlay IPs.
- Join-token comparison is constant-time (constant_time_eq over the stored hash, nodes.rs:591), and registration is blocked entirely when no join token is configured (nodes.rs:612-618), avoiding accidental open enrollment.
- Registration applies an SSRF guard on both private_address and address, rejecting reserved/dangerous IP ranges before persisting (nodes.rs:626-656), and both join paths mandate strict TLS for the inner register call with explicit comments explaining the MITM/join-token-theft risk (join.rs:128-130, 264-270).
- Node tokens used for control-plane↔agent auth are stored hashed (token_hash) plus AES-256-GCM-encrypted (token_encrypted) via EncryptionService rather than in plaintext (nodes.rs:658-668, node_service.rs:92,117).
- Persistent keepalive (25s) is set on peers for NAT traversal/liveness (lib.rs:241), and the compute-CIDR allocator enforces non-overlapping per-node CIDRs with a partial-unique DB index plus validate_with overlap checks (allocator.rs:236-242, config.rs:197-229).

## Secret & credential distribution to worker nodes

> I audited how secrets, env vars, S3/backup credentials, and the agent auth token flow from the control plane (temps serve) to worker nodes (temps agent) and how they are stored at rest on the worker. I traced the full path: workflow planner seals env/secrets into deployment_jobs.job_config (sensitive_envelope.rs, AES-256-GCM); the executor decrypts and builds a DeployRequest; RemoteNodeDeployer (temps-deployer/src/remote.rs) POSTs it to the agent's /agent/containers/deploy; the agent (temps-agent) creates the container via bollard, injecting environment_vars as Docker env and writing secrets to a 0700/0400 host dir bind-mounted ro at /run/secrets (temps-deployer/src/docker.rs). I also examined agent bearer-token auth (auth.rs), node registration/join token verification (handlers/nodes.rs, cli/commands/join.rs), token encryption at rest in the DB (node_service.rs), and S3 backup credential delivery (service_handlers.rs). Overall posture is mixed: at-rest handling is genuinely good (envelope encryption in DB, encrypted node tokens, file-based secrets with tight perms, strict TLS on the join/register and heartbeat control-plane calls, hashed+constant-time join token). The big gap is the data plane between control plane and agent: the agent serves plain HTTP with no TLS, nodes register their address as http://, so every deploy ships env vars, app secrets, and S3 creds in cleartext — security depends entirely on WireGuard (relay mode) or operator network hygiene (direct mode). Secondary gaps: name-based registration overwrite enables agent redirection/takeover with the shared join token, no per-node token rotation/revocation, and an unauthenticated Swagger UI on the agent.

### `secrets-1` — 🟠 HIGH — Control-plane → agent data plane is plaintext HTTP; env vars, secrets and S3 creds sent in cleartext

- **Verification:** ✅ confirmed via duplicate agentbnd-2 / exec-2 / transport-1 (plaintext HTTP) (verifier rate-limited)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/join.rs:138 and :288; crates/temps-agent/src/server.rs:367-384; crates/temps-deployer/src/remote.rs:29-61; crates/temps-deployer/src/lib.rs:136-146`
- **What's wrong:** At join time the node registers its agent address with an http:// scheme: `"address": format!("http://{}:{}", private_address.trim(), ...)` (join.rs:138, and identically :288 for relay mode). The agent server binds a plain `tokio::net::TcpListener` and serves with `axum::serve(listener, router)` — there is no TLS terminator anywhere in temps-agent (server.rs:367-384). The control plane's `RemoteNodeDeployer` uses the stored address verbatim, so although the reqwest client is built with `danger_accept_invalid_certs(insecure_tls_enabled())` (remote.rs:46), the actual transport is http://. The `DeployRequest` it POSTs carries `pub environment_vars: HashMap<String,String>` and `pub secrets: HashMap<String,String>` (lib.rs:139,146) in plaintext JSON. The same channel ships `ServiceBackupRequest.s3` (access_key_id / secret_key) for backups (service_handlers.rs:1237-1259). So all application env, all app secrets, and S3 credentials cross the wire unencrypted.
- **Scenario:** In Direct mode (`--private-address`) the control-plane↔agent link may traverse a shared VLAN, a cloud VPC, or the public internet with no encryption. Any on-path attacker (compromised switch, rogue host on the same subnet, cloud provider tap) passively captures every deployment's DB passwords, API keys, and S3 credentials, plus the agent bearer token (in the Authorization header), which then permits full container exec/takeover of the worker. Relay mode mitigates this via WireGuard, but Direct mode has no such protection and the code comments only promise TLS for the join/register call, not the data plane.
- **Recommendation:** Serve the agent API over TLS: have `temps join` provision a self-signed cert (or accept an operator cert), register the address as https://, and bind the agent with a rustls TLS acceptor (axum-server or tokio-rustls). Pin/verify the agent cert from the control plane (store its fingerprint at registration) instead of relying on the global `insecure_tls` escape hatch. At minimum, refuse to send secrets to an http:// node address unless the operator has explicitly acknowledged an out-of-band-encrypted link (e.g. WireGuard relay mode).

### `secrets-2` — 🟠 HIGH — Name-based node re-registration overwrites a live node's address+token, enabling agent redirection/takeover

- **Verification:** ✅ confirmed via duplicate enroll-1 / agentbnd-4 (name-collision takeover) (verifier rate-limited)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-deployments/src/services/node_service.rs:78-108; crates/temps-deployments/src/handlers/nodes.rs:585-689`
- **What's wrong:** `register()` looks up an existing node by NAME and, if found, blindly overwrites its security-critical fields: `active.token_hash = Set(...); active.token_encrypted = Set(...); active.address = Set(request.address); active.private_address = Set(request.private_address)` and sets status back to active (node_service.rs:86-95). The register handler authenticates only the shared cluster join token (nodes.rs:585-610) — there is no proof that the caller controls the node it is overwriting. Anyone with the join token can register `name = "prod-worker-1"` with `address = http://attacker:3100` and a token they chose.
- **Scenario:** An attacker who obtains the long-lived shared join token (it is non-expiring, single-secret, and travels on every join) re-registers under the name of an existing production worker, pointing `address` at an attacker-controlled host. The control plane's next remote deploy to that node sends the full DeployRequest — every env var and secret for that deployment — to the attacker, who returns a success envelope. This is silent cross-tenant/secret exfiltration plus deploy hijacking with only the join token, no operator session, and no agent compromise.
- **Recommendation:** Bind the agent token to the node identity and reject reconnection that changes address/token without proving control of the old token (e.g. require the existing token_hash to re-register, or require an operator-approved re-enroll). Make node identity a server-issued opaque node_id, not the operator-supplied name. Audit-log and alert on address/token changes during reconnection.

### `secrets-3` — 🟡 MEDIUM — No rotation or revocation path for worker agent tokens; deleted nodes keep valid credentials

- **Verification:** ✅ confirmed via duplicate enroll-5 / agentbnd-5 / exec-6 (no token rotation) (verifier rate-limited)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-deployments/src/services/node_service.rs (no rotate/revoke methods); crates/temps-agent/src/auth.rs:14-31; crates/temps-cli/src/commands/agent.rs:221-229`
- **What's wrong:** The agent's bearer token is a single static value baked into `AgentAuth::new(&config.token)` at startup (auth.rs:36 via server.rs) and loaded from `agent.json` (agent.rs:221-229). There is no API to rotate or revoke a node's agent token (grep for rotate/revoke/regenerate in nodes.rs and node_service.rs returns nothing). The token only changes by re-running `temps join`, which mints a fresh `generate_token()` and overwrites the row — but the worker process keeps serving the OLD token until manually restarted, and deleting the node from the DB does not invalidate the token the worker still honors.
- **Scenario:** After a suspected token leak (e.g. via the plaintext-HTTP exposure in secrets-1, a leaked backup of agent.json, or a process memory dump) an operator has no way to revoke just that worker's credential. The leaked token keeps granting container deploy/exec on the worker indefinitely. Conversely, deleting a node from the control plane leaves the worker still authenticating and heartbeating with a credential the platform believes is gone.
- **Recommendation:** Add a node-token rotation endpoint that pushes a new token to the agent (over the authenticated channel) and atomically updates token_hash/token_encrypted, plus a revocation that the agent honors (e.g. agent re-validates its token/identity against the control plane on a schedule and shuts down on 401). Restart/reload the AgentAuth in-process on rotation rather than requiring a manual restart.

### `secrets-5` — 🟡 MEDIUM — Shared, non-expiring join token is the only gate for enrolling a node that receives all deployment secrets

- **Verification:** ✅ confirmed via duplicate enroll-2 / wg-5 (shared join token) (verifier rate-limited)
- **Category:** security · **Confidence:** medium · **Location:** `crates/temps-core/src/app_settings.rs:413-420; crates/temps-deployments/src/handlers/nodes.rs:585-619`
- **What's wrong:** Cluster enrollment is gated by a single `join_token_hash: Option<String>` stored in MultiNodeSettings (app_settings.rs:414) with no expiry, no per-node scoping, no use-count limit, and no issued-at/rotation metadata. The handler verifies it with a hash + constant-time compare (nodes.rs:590-599) — which is good — but one shared secret enrolls unlimited nodes forever. A worker, once enrolled, is a deploy target that receives plaintext env/secrets for whatever workloads the scheduler places on it (deploy_image.rs node assignment).
- **Scenario:** The join token is copied into provisioning scripts, CI, and worker bootstrap (worker.sh). A single leak (committed script, shell history, log) lets an attacker enroll a rogue worker that the scheduler may then place real workloads on, handing the attacker live application secrets — with no way to expire or scope the leaked token short of rotating it for the entire fleet. Combined with secrets-2 it also enables name-overwrite takeover.
- **Recommendation:** Make join tokens single-use or short-TTL and per-enrollment (issue a one-time bootstrap token per node from the UI, consumed on register), record issued-at/used-at, and support invalidation. Optionally require operator approval to move a freshly-joined node into the schedulable pool before any secret-bearing workload lands on it.

### `secrets-4` — ⚪ LOW — Agent Swagger UI mounted without authentication, exposing the full worker API surface

- **Verification:** ✅ confirmed via duplicate transport-7 (agent Swagger unauth) (verifier rate-limited)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-agent/src/server.rs:107-115`
- **What's wrong:** All real routes are protected by `require_agent_auth` (server.rs:107), but the SwaggerUI is merged AFTER and outside the auth layer: `let swagger_ui = SwaggerUi::new("/swagger-ui").url("/api-docs/openapi.json", AgentApiDoc::openapi()); api_routes.merge(swagger_ui)` (server.rs:111-115). The inline comment even states 'no auth required so it's accessible for documentation'. The OpenAPI doc enumerates every privileged endpoint (container deploy, exec, terminal, service create/exec/backup with S3 creds).
- **Scenario:** Anyone who can reach the agent port (which, per secrets-1, may be on a shared/untrusted network and is bound to 0.0.0.0 in some configs) can fetch /api-docs/openapi.json and /swagger-ui without credentials, mapping the entire attack surface (exec, terminal, backup-with-S3) for a follow-on attack. It is information disclosure that directly aids exploitation of the other findings.
- **Recommendation:** Put the Swagger UI and openapi.json behind the same bearer-auth layer, or gate it behind an explicit `--enable-docs` debug flag that is off by default in production. Do not expose the agent API schema on an unauthenticated, network-reachable endpoint.

### `secrets-6` — ⚪ LOW — Deployment env vars (which routinely include connection strings/passwords) are injected as Docker env, readable via docker inspect and /proc/environ on the worker

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** security · **Confidence:** medium · **Location:** `crates/temps-deployer/src/docker.rs:1653-1657; crates/temps-deployments/src/jobs/deploy_image.rs:1095-1099`
- **What's wrong:** The `environment_variables` map is injected straight into the container's Docker env: `env: Some(... .map(|(k,v)| format!("{}={}", k, v)) ...)` (docker.rs:1653-1657). Only the separate `secrets` map gets file-based 0400 protection at /run/secrets. The deploy code itself acknowledges env values are sensitive — it truncates POSTGRES_URL before logging because it 'may contain password' (deploy_image.rs:1098-1099). Yet the same connection string is set as a plain container env var, visible to anyone who can run `docker inspect`, read `/proc/<pid>/environ`, or compromise any co-located container with host access on the worker.
- **Scenario:** On a multi-tenant or shared worker, an operator (or an attacker who gains read access to the Docker socket or host) runs `docker inspect <container>` and reads POSTGRES_URL, REDIS_URL, third-party API keys, etc., in cleartext for every deployed app. This is the standard env-vs-secret tradeoff, but because Temps auto-derives connection strings (with passwords) into env vars rather than secrets, sensitive material lands in the inspectable surface by default.
- **Recommendation:** Route auto-derived connection strings and any value detected as credential-like (URLs with embedded passwords, *_KEY, *_SECRET, *_TOKEN, *_PASSWORD) through the file-based /run/secrets path instead of Docker env, or document clearly that env vars are inspectable and steer users to mark sensitive values as secrets. At minimum, derive POSTGRES_URL-style values as secrets rather than env.

**✅ Strengths already in place (do NOT regress):**

- At-rest in the control-plane DB is well protected: env vars, remote env vars, secrets, and build args are envelope-encrypted with AES-256-GCM before being written into deployment_jobs.job_config, and only a *_keys list (names, not values) is stored in cleartext for debugging (crates/temps-deployments/src/services/sensitive_envelope.rs). Tests assert the plaintext value never appears in the serialized job_config.
- The worker agent token is never stored in plaintext in the control-plane DB — it is encrypted with the platform EncryptionService at registration (`encryption_service.encrypt(request.token.as_bytes())`, nodes.rs:660-668) and only decrypted in-memory at deploy time (deploy_image.rs:981-994).
- App secrets (the `secrets` map) are delivered to the worker as files, not env vars: written to a per-container host dir at mode 0700 with each file 0400, chowned to the image's resolved USER, and bind-mounted read-only at /run/secrets — so they are NOT visible via `docker inspect` or `/proc/environ` (crates/temps-deployer/src/docker.rs:1577-1592, 2170-2249). Secret key names are validated against path-escape characters.
- Agent bearer-token auth is implemented carefully: the token is SHA-256 hashed in memory and compared with a constant-time equality routine to avoid timing side-channels (crates/temps-agent/src/auth.rs:25-49), and the node-side join token is verified the same way (constant_time_eq in handlers/nodes.rs:591).
- The security-critical control-plane-bound calls (worker→control-plane heartbeat carrying the token, and CLI join/register carrying the join token) deliberately use strict TLS with no insecure opt-in for CLI binaries — the code comments explicitly reason about MitM token theft and refuse the insecure_tls escape hatch on these paths (crates/temps-agent/src/server.rs:146-158; crates/temps-cli/src/commands/join.rs:124-130, 264-278).
- The join token is generated with 32 bytes of randomness and hex-encoded (256-bit), and the per-node agent token is likewise 32 random bytes (crates/temps-cli/src/commands/join.rs:368-374) — adequate entropy with no predictable/derived secrets.
- Node registration applies an SSRF guard on both private_address and the user-supplied agent address, rejecting reserved/dangerous ranges before they are persisted and later used to build deploy/health URLs (crates/temps-deployments/src/handlers/nodes.rs:623-656).
- The agent.json that holds the worker's own agent token is written with directory 0700 and file 0600 permissions (crates/temps-cli/src/commands/join.rs:90,101), so the local credential is not world-readable on the worker.

## Node lifecycle, health, failover & scheduler resilience

> I audited the multi-node lifecycle/health/failover/scheduler path in temps-deployments: node_health_check.rs (the 60s control-plane health job + failover), node_service.rs (register/heartbeat/mark_offline/drain/remove/reconcile), node_scheduler.rs (replica placement), the node HTTP handlers (register/heartbeat/drain/remove), the periodic-job wiring in temps-cli serve/console.rs, the WireGuard peer API (temps-wireguard), the overlay CIDR allocator (temps-network), and the node_route_state/node_dns_state entities. Overall the happy-path lifecycle is sound: token-authenticated heartbeats, a stale-heartbeat→offline→failover loop, drain orchestration, and ghost-record reconciliation all exist and are unit-tested. The weaknesses are in the failure boundaries: node removal leaks WireGuard peers and overlay CIDR allocations (dead code paths exist but are never called), failover redeploy has no retry and can strand workloads, a recovered node's still-running containers are never stopped after its workloads were failed over (duplicate live containers), there is no leader election so the failover loop double-fires under the ADR-017 split topology, and reconnection/heartbeat carries no replay/rotation protection on a never-expiring shared token. These are reliability + secret-lifecycle gaps rather than the happy path being broken.

### `lifecycle-1` — 🟠 HIGH — Node removal never revokes the WireGuard peer or frees the overlay CIDR (resource + trust leak)

- **Verification:** ✅ confirmed via duplicate enroll-6 / wg-3 (removal doesn't revoke WG/CIDR) (verifier rate-limited)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-deployments/src/handlers/nodes.rs:1720-1762 (admin_remove_node) and crates/temps-deployments/src/services/node_service.rs:336-348 (remove); cf. crates/temps-wireguard/src/lib.rs:262 (remove_peer, uncalled) and crates/temps-network/src/allocator.rs:270 (release, uncalled)`
- **What's wrong:** admin_remove_node only calls node_service.remove(node_id), which does `nodes::Entity::delete_by_id(node_id)` and nothing else. The WireGuard API exposes `pub async fn remove_peer(...)` (temps-wireguard/src/lib.rs:262) and the overlay allocator exposes `async fn release(&self, node_id)` (allocator.rs:270), but `grep -rn remove_peer` / `.release(` show NEITHER is ever invoked from the node-removal path (the only release() callers are the unrelated agents sandbox_registry). The node row delete cascades node_route_state/node_dns_state (ON DELETE CASCADE) but the mesh peer entry and the compute_cidr stay.
- **Scenario:** An operator removes a worker (e.g. decommission, or a node they no longer trust). The removed node's WireGuard public key remains an allowed peer in the mesh, so a host still holding that node's key/token can keep tunneling into the overlay and reaching other nodes' agents — a removed/rotated-out node is not actually cut off. Separately, every join/remove cycle permanently consumes a /24 from the cluster pool (allocate_overlay_cidr on join, never released), so churn eventually exhausts the CIDR pool and new joins fail with no obvious cause.
- **Recommendation:** In admin_remove_node, before/after node_service.remove(): call allocator.release(node_id) to free the compute_cidr, and remove the node's wg_public_key from the mesh via WireGuardService::remove_peer (and propagate peer removal to remaining nodes' agents via the existing network_sync/peers mechanism). Make removal best-effort-logged but ensure the peer is gone. Add a test asserting release+remove_peer are called on node removal.

### `lifecycle-2` — 🟠 HIGH — A recovered node's orphaned containers are never stopped after failover (duplicate live replicas)

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-deployments/src/services/node_service.rs:473-510 (reconcile_containers) and crates/temps-deployments/src/handlers/nodes.rs:824, 837-869 (heartbeat reactivation + reconcile)`
- **What's wrong:** reconcile_containers is one-directional: it only marks DB records as deleted when the container is NOT in the agent's reported list (`if !actual_container_ids.contains(&container.container_id)`). There is no reverse path that tells the agent to stop containers that are running on the node but no longer tracked by the CP. Meanwhile node_heartbeat reactivates an offline node (handlers/nodes.rs:824 `was_offline`, service line 156 `if node.status == "offline" { active.status = Set("active") }`). A grep of temps-agent shows no orphan-prune path either.
- **Scenario:** Node N has a network blip > 90s. check_node_health marks N offline and failover_offline_nodes redeploys its workloads onto healthy nodes (needs_redeploy branch). N's connectivity returns; its agent and containers were never actually killed. On the next heartbeat N flips back to active and reports its still-running containers. reconcile only removes ghost DB rows — it does NOT stop the now-duplicate app containers on N. Two live copies of the same service serve traffic (double writes, double cron, double background workers), and the proxy may even route to the stale copy.
- **Recommendation:** After a node returns from offline (was_offline==true), reconcile in BOTH directions: for any temps-managed container the agent reports that the CP no longer has an active record for (failed over / superseded), issue stop_container to the agent. Alternatively, on offline→active transition, treat the node as needing a clean re-sync: stop all containers the CP didn't (re)assign to it. Add an integration test for the failover-then-recover duplicate-container scenario.

### `lifecycle-3` — 🟠 HIGH — Failover redeploy has no retry and silently strands the workload on failure

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-deployments/src/jobs/node_health_check.rs:135-159 (failover redeploy branch) and crates/temps-deployments/src/services/services.rs:1087-1120 (redeploy_environment triggers a full pipeline)`
- **What's wrong:** In failover_offline_nodes, the needs_redeploy() branch calls deployment_service.redeploy_environment(...). On Err it only `tracing::error!`s and continues — there is no retry, no backoff, no alert, no re-queue. redeploy_environment (services.rs:1118) calls trigger_pipeline, i.e. a full rebuild/clone pipeline, not a fast re-pull of the existing image. Once a node is marked offline, check_node_health filters `status == "active"` so it will NEVER re-emit that node id on later passes, meaning the 60s loop will not retry the failed failover.
- **Scenario:** A node dies during an incident that also makes the git source momentarily unreachable (or the build queue is saturated). The single failover redeploy attempt fails, is logged, and dropped. Because the node stays 'offline' and is never re-selected by check_node_health, no further failover is attempted. The environment with all replicas on the dead node stays DOWN indefinitely until a human notices — exactly when availability matters most.
- **Recommendation:** Wrap the failover redeploy in RetryConfig with bounded exponential backoff, and on persistent failure record a durable 'pending failover' marker (e.g. a column/row) that a reconciler retries, plus raise an alert/notification. Prefer image-reuse redeploy over full pipeline for failover so it doesn't depend on the source being reachable. Add a test for redeploy failure → retry/marker.

### `lifecycle-4` — 🟡 MEDIUM — No leader election: failover loop runs on every console/control instance → concurrent double-redeploy

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** reliability · **Confidence:** medium · **Location:** `crates/temps-cli/src/commands/serve/console.rs:1775-1804 (tokio::spawn 60s health+failover loop, no coordination)`
- **What's wrong:** The health-check+failover task is an unconditional `tokio::spawn` started in console startup. There is no leader election, advisory lock, or PG-NOTIFY coordination guarding it. MEMORY/ADR-017 describe a split proxy/console topology and HA where multiple control-plane processes can run. mark_offline and the redeploy are not guarded by any 'only one runner' check, and the offline transition is a plain read-then-write (find node, set status) with no optimistic-concurrency/transaction guard.
- **Scenario:** Under the split or HA topology, two console processes each tick their 60s timer. Both observe the same stale node, both call mark_offline, and — critically — both call redeploy_environment / retire_containers for the same affected deployments at nearly the same time. This produces duplicate redeploy pipelines and racey container retirement (TOCTOU between affected_deployments snapshot and retire/redeploy), wasting build capacity and potentially flapping the environment.
- **Recommendation:** Gate the health/failover loop behind a single-runner mechanism: a Postgres advisory lock (pg_try_advisory_lock) or the existing leader/role check so only one instance runs it. Make the offline transition + failover idempotent with a transactional compare-and-set on status and a per-(deployment) in-progress guard so a concurrent runner is a no-op.

### `lifecycle-5` — 🟡 MEDIUM — Node token is a never-expiring shared secret with no rotation/replay protection on heartbeat/reconnect

- **Verification:** ✅ confirmed via duplicate enroll-5 / exec-6 (node token never expires) (verifier rate-limited)
- **Category:** security · **Confidence:** medium · **Location:** `crates/temps-deployments/src/handlers/nodes.rs:799-835 (node_heartbeat) and 560-687 (register_node reconnection) and crates/temps-deployments/src/services/node_service.rs:84-108 (register reconnect overwrites token/keys by name match)`
- **What's wrong:** Heartbeats authenticate by comparing sha256(token) to nodes.token_hash with constant_time_eq (good), but the token never expires, has no nonce/timestamp, and there is no rotation path. register() treats any registration whose `name` matches an existing node as a reconnection and overwrites token_hash, address, private_address, wg_public_key and edge_public_key, then sets status=active — gated only by the single cluster-wide join_token (handlers/nodes.rs:585-619).
- **Scenario:** Anyone who learns the one cluster join token (it is shared across all workers and stored as a single settings hash) can POST /internal/nodes/register with the name of an existing worker and hijack that node's identity: replace its address/agent URL and WireGuard key, flip it active, and from then on the control plane will deploy that node's workloads to attacker-controlled infrastructure (deploy of attacker payload / data-plane redirection). There is no per-node enrollment secret or rotation to contain a leaked token.
- **Recommendation:** Issue a per-node credential at first registration and require it (not just the shared join token) for reconnection; reject a name-collision reconnect unless it presents the existing node's secret. Add token rotation + expiry, and add a replay-resistant heartbeat (signed timestamp/nonce). At minimum, log/alert and require admin approval when a reconnect changes a node's address or wg_public_key.

### `lifecycle-6` — 🟡 MEDIUM — Scheduler silently falls back to the control plane when all pinned target nodes are unavailable

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-deployments/src/services/node_scheduler.rs:164-190 (target_node_ids filter → empty → return vec![Local; replica_count])`
- **What's wrong:** When a deployment pins to specific nodes via target_node_ids and none of them are currently in list_active() (offline/draining/stale heartbeat), eligible_nodes is empty and the scheduler returns `Ok(vec![NodeAssignment::Local; replica_count])` after only a `tracing::warn!`. The pin is silently ignored and the workload is placed on the control-plane host instead.
- **Scenario:** An operator pins a workload to a labeled/dedicated node pool (e.g. GPU nodes, or a tenant-isolated node, or a region for data-residency). Those nodes briefly drop below the heartbeat threshold during a deploy. The scheduler quietly schedules the replicas onto the control-plane node — violating the placement constraint (wrong region/isolation/hardware) with no error surfaced to the caller. For data-residency or tenant-isolation pins this is a compliance/isolation break, not just a perf issue.
- **Recommendation:** For explicit target_node_ids (a hard constraint), do not fall back to Local — return a typed NodeError (e.g. NoEligibleTargetNodes) so the deploy fails loudly and can be retried, or queue it until a target node is healthy. Reserve Local-fallback for the unpinned/best-effort case only. Add a test asserting an error (not Local) when all pinned targets are down.

### `lifecycle-7` — 🟡 MEDIUM — check_drain_completion is implemented and intended for the 60s loop but never wired in — drains can hang in 'draining' forever

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-deployments/src/jobs/node_health_check.rs:79-92 (check_drain_completion) vs crates/temps-cli/src/commands/serve/console.rs:1781-1801 (loop only calls check_node_health + failover_offline_nodes)`
- **What's wrong:** node_health_check.rs documents check_drain_completion as 'designed to be called after check_node_health in the same periodic job (every 60 seconds)', and check_all_drains() transitions draining→drained when containers reach zero. But the actual spawned loop in console.rs only calls check_node_health and failover_offline_nodes — check_drain_completion / check_all_drains is NOT invoked there. The only other driver is admin_drain_status (handlers/nodes.rs:1789), which requires a human to poll GET /drain.
- **Scenario:** An operator drains a node and walks away without polling the drain-status endpoint. Containers finish migrating and the node has zero containers, but because nothing periodically calls check_all_drains, the node stays in 'draining' (never auto-transitions to 'drained'). It is excluded from scheduling (list_active filters status='active') yet is never reported drain-complete/safe-to-remove, leaving capacity stranded and the lifecycle stuck until someone manually hits the status endpoint.
- **Recommendation:** Call check_drain_completion(&node_service) (which wraps check_all_drains) inside the existing 60s loop right after check_node_health, as the module docs already intend. Add coverage that a draining node with zero containers auto-transitions to 'drained' via the periodic job.

### `lifecycle-8` — 🟡 MEDIUM — Container retirement during drain/failover is best-effort: agent stop failures leave running containers while DB marks them removed

- **Verification:** 🔶 finder-reported, NOT re-verified (verifier rate-limited)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-deployments/src/handlers/nodes.rs:1586-1627 (drain: stop_container failures only warn, then retire_containers_on_node soft-deletes regardless) and crates/temps-deployments/src/services/node_service.rs:515-545 (retire_containers_on_node only soft-deletes DB rows)`
- **What's wrong:** In admin_drain_node, if create_remote_deployer fails or remote_deployer.stop_container errors, the code only `warn!`s ('will still retire') and then unconditionally calls retire_containers_on_node, which sets deleted_at + status='removed' on the DB rows. The failover path (node_health_check.rs:160-185) likewise calls retire_containers_on_node with no agent stop at all. The DB now says the containers are gone, but the actual Docker containers may still be running on the node.
- **Scenario:** During a drain where the agent is reachable-but-flaky (or during failover where the node is unreachable but later recovers), the control plane records containers as removed and stops routing to them, but the real containers keep running on the worker — consuming resources, holding ports/DB connections, running crons, and (combined with finding lifecycle-2) producing duplicate live replicas once the node is reachable again. Operators see 'drained/removed' in the UI while workloads are still live.
- **Recommendation:** Make retirement converge: when stop_container fails or the node is unreachable, do not treat the container as fully removed — mark it 'pending-stop' and have the reconciler/agent re-attempt stop until Docker confirms it's gone, only then soft-delete. On node recovery, force-stop any container the CP marked removed. Surface partial-drain state instead of reporting drain_complete prematurely.

**✅ Strengths already in place (do NOT regress):**

- Heartbeat authentication is sound: tokens are compared as sha256 hashes using constant_time_eq (handlers/nodes.rs:816, 911), avoiding timing side-channels, and the plaintext token is encrypted at rest via EncryptionService (register_node:660).
- Node registration enforces a configured join token (rejects all registrations when none is set) and applies an SSRF guard (validate_node_private_address) to both private_address and the agent address before persisting them (handlers/nodes.rs:585-656).
- The stale-heartbeat threshold (90s) is deliberately > the 30s heartbeat interval and bounded ≤300s, with a compile-time test asserting the invariant (node_health_check.rs:220-225), reducing flapping.
- check_node_health only returns nodes that transition active→offline (it filters status='active'), so the 60s failover loop does not repeatedly re-fire redeploys against an already-offline node — failover is effectively edge-triggered, not level-triggered.
- Failover is replica-aware: needs_redeploy() (node_service.rs:563) distinguishes 'all replicas on dead node → full redeploy' from 'healthy replicas elsewhere → just retire stale containers', avoiding unnecessary rebuilds when capacity remains.
- node_route_state / node_dns_state use the node id as PK with ON DELETE CASCADE to nodes, so per-node route/DNS applied-state is cleaned up automatically when a node row is deleted (node_route_state.rs:31-40, node_dns_state.rs:39-48).
- heartbeat() preserves managed states: it only reactivates a node from 'offline' and does not clobber 'draining'/'drained' (node_service.rs:154-158), with explicit tests (test_heartbeat_preserves_draining_status, test_heartbeat_reactivates_offline_node).
- reconcile_containers provides ghost-record cleanup (DB rows whose containers no longer exist on the node are marked removed) on agent reconnect, preventing the proxy from routing to dead containers after a crash (node_service.rs:473-510).
- The scheduler degrades gracefully under load: nodes over the max-load threshold are excluded, but if ALL exceed it the limit is relaxed so scheduling still proceeds rather than failing hard (node_scheduler.rs:212-245).

## Remote container exec & command surface

> I audited the cross-node exec/command surface: the control-plane handlers (crates/temps-deployments/src/handlers/container_exec.rs), the worker agent's HTTP/WebSocket exec endpoints (crates/temps-agent/src/handlers.rs exec_container/terminal_container, service_handlers.rs service_exec), the agent auth middleware (crates/temps-agent/src/auth.rs), the remote-exec service glue (temps-deployments/src/services/services.rs, temps-deployer/src/remote.rs), token generation at join (temps-cli/src/commands/join.rs), and the agent server bind/TLS (temps-agent/src/server.rs) plus worker.sh.  Overall posture: the CP-facing exec/terminal handlers are correctly gated by `permission_guard!(auth, ContainersExec)` and verify the target container belongs to the requested project/environment via get_container_detail before routing to a node, which closes the obvious cross-tenant-via-CP path. The agent uses a 32-byte random bearer token, SHA-256 hashed, compared in constant time. Timeouts and idle limits exist on most exec paths. However the agent itself is a fully trusted, tenant-blind command executor: a single node-wide bearer token authorizes arbitrary `docker exec`/PTY/`service_exec` against ANY container_id on the host with no project scoping, the transport is hardcoded plaintext http:// (no TLS in the agent at all), there is no audit logging on exec/terminal despite the project mandate, a documented `container_exec_enabled` gate is not actually enforced, `service_exec` runs as an arbitrary `user` with no timeout, and the node token has no rotation/revocation/expiry. The security of the entire data plane therefore rests almost entirely on the WireGuard mesh being present and the token never leaking.

### `exec-1` — 🟠 HIGH — Agent exec endpoints are tenant-blind: one node token execs ANY container on the host

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-agent/src/handlers.rs:451-504 (exec_container), :582-597 (terminal_container); crates/temps-agent/src/service_handlers.rs:650-710 (service_exec)`
- **What's wrong:** The agent's exec_container/terminal_container/service_exec handlers take the target from the request path/body and call `docker.create_exec(&container_id, ...)` / `create_exec(&request.container_name, ...)` with NO check that the container belongs to the caller's project/tenant. The agent has no concept of tenancy — `pub async fn exec_container(State(state), Path(container_id), Json(request))` execs whatever container id is supplied. Ownership is only verified one hop earlier on the CP (`get_container_detail(project_id, environment_id, container_id)` in container_exec.rs:81). The single node-wide bearer token (auth.rs) is the only thing between a caller and arbitrary exec into every container on the node — including other tenants' workloads co-located on the same worker.
- **Scenario:** An attacker who obtains the node bearer token (leaked from nodes.token_encrypted, a compromised CP, an SSRF that can reach the agent, or any host with WG access) POSTs to `/agent/containers/<victim-tenant-container>/exec` with `["sh","-c","cat /app/.env"]` and reads secrets / runs code in every co-tenant container on that worker. The CP-side project ownership check is fully bypassed because the agent is reached directly.
- **Recommendation:** Make the agent enforce tenancy itself rather than trusting the path. Options: have the CP mint short-lived, container-scoped exec capability tokens (signed, naming the exact container id + TTL + operation) that the agent verifies, OR have the agent label-check the target container's `temps.project_id`/owner against a per-request scope claim before create_exec. Do not rely solely on the CP to scope a node-wide credential.

### `exec-2` — 🟠 HIGH — Agent transport is hardcoded plaintext HTTP with no TLS — exec/PTY/token in cleartext off the WG mesh

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/join.rs:138 (`"address": format!("http://{}:{}", ...)`); crates/temps-agent/src/server.rs:367-384 (plain TcpListener, no rustls); crates/temps-deployer/src/remote.rs:30,68 (agent_url, bearer_auth over http)`
- **What's wrong:** At join the node registers its CP-facing address as a hardcoded `http://{private_address}:{port}` (join.rs:138) — never https. The agent server binds a plain `tokio::net::TcpListener` and serves over `axum::serve` with no TLS anywhere (the crate has no rustls/axum-server dependency at all). Every CP→agent call — deploy payloads, `exec_command` bodies, the terminal WebSocket PTY bytes, and the `Authorization: Bearer <node-token>` header (remote.rs:72, container_exec.rs:311) — crosses the wire in cleartext. Confidentiality depends entirely on WireGuard being in front. `validate_node_private_address` (handlers/nodes.rs:374-379) explicitly ALLOWS public IPs ('Workers that use public IPs ... are intentionally allowed'), so a Direct-mode node on a public/LAN address with no WG exposes a plaintext exec channel and a sniffable bearer token.
- **Scenario:** A Direct-mode worker is registered with a routable private/public IP and the operator sets --agent-address to that interface so the CP can reach it (loopback default won't work cross-host). Anyone on that network path passively captures the static bearer token and the interactive terminal/exec traffic, then replays the token to exec into any container (see exec-1). The token never rotates, so the capture is permanently valid.
- **Recommendation:** Support TLS on the agent (rustls server cert, pin via CP), register https:// addresses, and/or refuse to operate the exec/terminal channel unless the peer is reached over the WG interface. At minimum warn loudly and document that Direct mode without WG transmits exec traffic and the node token in cleartext.

### `exec-3` — 🟡 MEDIUM — No audit logging on container exec or terminal sessions

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-deployments/src/handlers/container_exec.rs:67-235 (exec_command and container_terminal — no audit_service call anywhere)`
- **What's wrong:** Both the one-shot exec handler and the interactive terminal WebSocket handler perform a privileged, code-execution-equivalent operation but emit no audit log. There is only a `tracing::info!` line. The project's own CLAUDE.md mandates 'Add audit logging for all write operations' and 'Add audit logging for all write operations (CREATE, UPDATE, DELETE)'. Exec is the single most security-sensitive write action in the system, yet `grep audit` over container_exec.rs returns nothing. The handler doesn't even take `Extension<RequestMetadata>` to capture IP/user-agent.
- **Scenario:** An insider or compromised account with ContainersExec opens a terminal into a production container and exfiltrates data or plants a backdoor. With no audit record of who exec'd into which container/project at what time, incident response cannot reconstruct the action or attribute it — there is no forensic trail for the highest-privilege operation in the platform.
- **Recommendation:** Emit an audit log on every exec_command and container_terminal invocation (and on the remote variants) capturing user_id, project_id, environment_id, container_id, node_id, the command (for one-shot), and source IP/UA via RequestMetadata, before performing the operation.

### `exec-4` — 🟡 MEDIUM — Documented `container_exec_enabled` per-project gate is never enforced

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-deployments/src/handlers/container_exec.rs:1-5 (doc: 'the project must have container_exec_enabled set to true'); entity at crates/temps-entities/src/deployment_config.rs:284 (default false at :423)`
- **What's wrong:** The module doc comment asserts exec/terminal 'require the ContainersExec permission and the project must have container_exec_enabled set to true.' The flag exists on the deployment config entity and defaults to false, but neither exec_command nor container_terminal ever reads it — there is no `if !...container_exec_enabled { return ... }` anywhere in the handler (confirmed: the only occurrence of the string in this file is the doc comment). The intended defense-in-depth opt-in is silently absent, so exec works on every project regardless of the operator's setting.
- **Scenario:** An operator deliberately leaves container_exec_enabled = false (the default) on sensitive projects believing exec is disabled, as the docs and UI toggle imply. A user with the ContainersExec permission still opens a shell into those containers because the flag is never checked, defeating the operator's stated isolation policy.
- **Recommendation:** Enforce the flag: fetch the effective deployment config for the project/environment and return 400/403 when container_exec_enabled is false, in BOTH exec_command and container_terminal, before any docker/agent call. Add a test asserting the disabled case returns the error.

### `exec-5` — 🟡 MEDIUM — service_exec runs arbitrary command as arbitrary user with no timeout

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-agent/src/service_handlers.rs:650-693 (user: request.user.as_deref(), cmd from request.command, attached collection at :739-762 with no tokio::time::timeout)`
- **What's wrong:** Unlike exec_container (which caps timeout at 300s, handlers.rs:469/511), the agent's service_exec has NO timeout: it builds an exec with caller-supplied `cmd`, caller-supplied `environment`, and a caller-supplied `user` (`user: request.user.as_deref()` — so it can run as root in the service container), then streams output in an unbounded `while let Some(chunk) = output.next().await` loop with no time bound. A command that never terminates or produces unbounded output holds the agent task and accumulates stdout/stderr into in-memory Strings indefinitely.
- **Scenario:** A control-plane-triggered service operation (or anyone with the node token) invokes /agent/services/exec with a long-running or high-output command (e.g. a hung psql or `yes`). The agent task never completes, the stdout String grows until the agent OOMs, and there is no upper bound or cancellation — a reliability/DoS gap on the worker. The arbitrary `user` field also broadens the blast radius beyond the container's default user.
- **Recommendation:** Wrap service_exec output collection in a bounded tokio::time::timeout like exec_container does, cap accumulated stdout/stderr size, and constrain/validate the `user` field (or drop it) so callers can't escalate to root inside the service container.

### `exec-6` — 🟡 MEDIUM — Node agent token has no rotation, revocation, or expiry

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/join.rs:121,368-374 (generate_token, used once at join); crates/temps-agent/src/auth.rs:18-30 (single static AgentAuth.token_hash for process lifetime)`
- **What's wrong:** The node bearer token is generated once at `temps join` (32 random bytes, hex) and baked into the agent process via AgentAuth::new at startup. It is a long-lived, never-expiring shared secret stored plaintext in ~/.temps/agent.json on the worker and encrypted in nodes.token_encrypted on the CP. There is no rotation path, no revocation list, and the agent re-hashes the same static token for every request forever. Combined with exec-1 (token = full-node exec) and exec-2 (token sniffable in cleartext off WG), a single leak is a permanent node compromise.
- **Scenario:** The token leaks (host backup, log, disk forensics, MITM per exec-2, or a CP DB read of token_encrypted plus the encryption key). The operator has no way to rotate or revoke it short of fully re-provisioning the node; the attacker retains arbitrary exec on every container on that worker indefinitely.
- **Recommendation:** Add a token rotation endpoint/flow (CP issues a new token, agent reloads it, old hash invalidated) and a revocation mechanism so a compromised node can be cut off without re-imaging. Consider short-lived, CP-signed capability tokens for exec specifically (ties into exec-1).

### `exec-7` — ⚪ LOW — Agent terminal WebSocket has no authn before docker exec is created; relies solely on Authorization header reaching the upgrade

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** medium · **Location:** `crates/temps-agent/src/handlers.rs:582-597 (terminal_container) + server.rs:54-57,107 (route under require_agent_auth layer)`
- **What's wrong:** The terminal WebSocket route is mounted under the bearer-auth middleware (server.rs:107), and the CP proxy sends the Authorization header on the WS upgrade request (container_exec.rs:309-318). That is the only gate. The agent handler itself does no per-container scope check before spinning up an interactive root-capable PTY (`handle_terminal_session` execs bash/sh, handlers.rs:607-631). This is the WebSocket manifestation of exec-1 but called out because WS auth is easy to get subtly wrong (browsers can't set Authorization on native WS; here it works only because the CP, not a browser, dials the agent). Any future client that dials the agent WS directly without the header, or a middleware ordering regression, would expose an unauthenticated root shell.
- **Scenario:** A refactor that moves the terminal route outside the auth layer, or a client that connects to the agent WS without the bearer header, yields an unauthenticated interactive shell into arbitrary containers. The handler provides no second line of defense.
- **Recommendation:** Add an explicit token/scope check inside terminal_container itself (defense in depth) and a test asserting the WS upgrade is rejected without a valid token, so auth never depends solely on middleware ordering plus the CP being the only caller.

### `exec-8` — ⚪ LOW — reqwest client for agent calls can accept invalid TLS certs globally via insecure_tls toggle

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** medium · **Location:** `crates/temps-deployer/src/remote.rs:40-56 (`.danger_accept_invalid_certs(temps_core::tls::insecure_tls_enabled())`)`
- **What's wrong:** The RemoteNodeDeployer HTTP client is built with `.danger_accept_invalid_certs(temps_core::tls::insecure_tls_enabled())`. If an operator flips the global insecure_tls setting (intended for self-signed CP certs), the CP→agent client will also accept any cert for the agent endpoint. Combined with exec-2 (agent registers http:// today), the moment someone fronts the agent with TLS, this toggle would silently disable cert validation on the exec/deploy channel, enabling an active MITM to impersonate the agent and capture/inject exec traffic.
- **Scenario:** An operator enables insecure_tls to tolerate a self-signed control-plane cert, unaware it also weakens agent connections. An on-path attacker presents a forged cert for the worker's agent endpoint; the CP accepts it and sends exec commands and the bearer token to the attacker, who relays or tampers with container exec.
- **Recommendation:** Scope the insecure_tls toggle so it does not apply to node/agent connections, or require a separate explicit per-node opt-in. Default the agent channel to strict verification (with a CP-pinned cert) independent of the global self-signed-CP convenience flag.

**✅ Strengths already in place (do NOT regress):**

- CP-side exec and terminal handlers are gated by `permission_guard!(auth, ContainersExec)` (container_exec.rs:71,243) — the permission system is correctly applied at the control plane.
- Container ownership is verified at the CP before routing to a node: get_container_detail(project_id, environment_id, container_id) is called and the DB-record container_id (not the client-supplied one) is used for the actual exec (container_exec.rs:80-95, 250-263), preventing the obvious cross-project exec via the CP API.
- One-shot exec enforces a timeout capped at 300s on both the CP-local and agent paths (container_exec.rs:97, handlers.rs:469/511), bounding runaway commands.
- Interactive terminal sessions have a 15-minute idle timeout on both the CP-local (container_exec.rs:469) and agent (handlers.rs:689) loops, preventing indefinitely-held idle PTYs.
- Agent bearer-token verification hashes with SHA-256 and compares in constant time with a length check first (auth.rs:38-49), avoiding token length/timing side channels, and rejects empty/missing/non-Bearer headers.
- The node agent token is 32 bytes of randomness (join.rs:369-374) — adequate entropy — and is stored encrypted at rest on the CP (nodes.token_encrypted, decrypted via EncryptionService only when needed, services.rs:319-333).
- Commands are passed as an explicit argv vector to bollard create_exec (no shell string interpolation by Temps), so there is no command/argument injection introduced by the platform itself — the array is handed straight to Docker's exec API.
- Node private_address registration is validated to reject loopback, link-local, multicast, broadcast and unspecified ranges before persistence (handlers/nodes.rs:374-399, validate_node_private_address), reducing SSRF/abuse via crafted node addresses.

## Transport security & relay trust (Direct vs Relay)

> I audited the control-plane↔agent transport and the trust placed in the relay (api.temps.sh) across crates/temps-cli/src/commands/join.rs, crates/temps-agent/ (server.rs, auth.rs, network_sync.rs, route_sync_client.rs), crates/temps-deployer/src/remote.rs, crates/temps-network/ (config.rs, linux/vxlan.rs), crates/temps-core/src/tls.rs, and crates/temps-deployments/src/handlers/nodes.rs.  Posture: The application-layer authentication of the control↔agent channel is solid — every agent endpoint requires a bearer token (SHA-256 hashed, constant-time compared), every worker→CP call (register/heartbeat/peers/routes) is bearer-authenticated, and the CLI deliberately refuses to honor the server-side insecure_tls opt-in, so join-token-carrying registration is always strict-TLS. SSRF guards exist on registered node addresses. However the *transport confidentiality/integrity* story is weak: the agent HTTP server is plaintext-only (no TLS termination), and the address the control plane uses to reach it is always built as http://, so all control-plane→agent traffic (container deploys, exec, log/stat streaming, S3-credential delivery) is cleartext that depends entirely on a correctly-configured WireGuard tunnel (relay mode) or the operator's own network (direct mode). In Direct mode there is no WireGuard at all, so a misconfigured underlay means full cleartext over whatever network the operator supplied. The relay is trusted on a TOFU basis for the control plane's WG pubkey, endpoint and URL — and in relay mode the relay even mints/relays the worker's agent_token, so a malicious or compromised relay can impersonate the worker, MITM the mesh bootstrap, or point the worker at an attacker-controlled control plane. The default VXLAN overlay carrying inter-node container traffic is unencrypted UDP over the underlay (which may be a public IP). A server-side insecure_tls toggle can globally disable cert verification on outbound clients including the remote deployer.

### `transport-1` — 🟠 HIGH — Control-plane→agent data plane is plaintext HTTP; no agent-side TLS, address hardcoded http://

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-agent/src/server.rs:384 (axum::serve, no TLS); crates/temps-cli/src/commands/join.rs:138 and 288 ("address": format!("http://{}:{}", ...)); crates/temps-deployer/src/remote.rs:68 (url = format!("{}{}", self.agent_url, path))`
- **What's wrong:** The agent server is started with `axum::serve(listener, router)` — plain HTTP, no TLS acceptor anywhere in temps-agent. The address the control plane stores and later dials is always built as `http://` by `temps join` (join.rs:138 direct, join.rs:288 relay). `RemoteNodeDeployer` honors `https://` if it were present (remote.rs:44-46) but the registered URL is never https. Therefore every control-plane→agent call — `deploy_container`, `exec_container`, `terminal`, `get_container_logs`, `import_image`, and `service_handlers` (which carry deploy payloads/env) — is unauthenticated-at-transport cleartext. Confidentiality and integrity rely 100% on a correctly-configured WireGuard tunnel or the operator's private network; TLS provides no defense-in-depth.
- **Scenario:** An attacker who can observe or tamper with traffic on the path between control plane and agent (a compromised intermediate host, a incorrectly scoped cloud VPC, a relay/WG endpoint that terminates and re-originates, or a direct-mode deployment whose 'private' link actually traverses shared infrastructure) reads the bearer token off the wire and/or rewrites a deploy request to launch an attacker image, or reads exec output / injected secrets (env vars, S3 creds). Because the only protection is the network layer, any WG/underlay misconfiguration silently downgrades the entire data plane to cleartext with no error surfaced.
- **Recommendation:** Terminate TLS on the agent (serve over rustls with a per-node cert minted at join, or pin a self-signed cert fingerprint the control plane records at registration) and register the agent address as https://. At minimum, transmit a cert fingerprint in the register payload and have RemoteNodeDeployer pin it, so the channel is authenticated independent of the network layer. Treat WireGuard as defense-in-depth, not the sole confidentiality control.

### `transport-2` — 🟠 HIGH — Direct mode has neither TLS nor WireGuard — fully cleartext, trust delegated to operator network with no enforcement

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/join.rs:108-182 (join_direct); join.rs:138 ("address": http://{private_address}:{port}); join.rs:27-30 (--private-address doc: "requires user-managed networking")`
- **What's wrong:** `join_direct` skips the entire WireGuard path and registers the agent with a bare `http://{private_address}:{port}` URL. The TLS-to-control-plane comment at join.rs:124-130 only protects the *registration* call to the control plane; the resulting control-plane→agent channel has no TLS and no WireGuard. The agent itself binds `0.0.0.0:3100` by default (temps-agent/src/lib.rs:368), so the plaintext, bearer-token-protected agent API is exposed on every interface, while the only confidentiality guarantee is the assumption that the operator placed both ends on a trusted private network. Nothing validates that assumption.
- **Scenario:** An operator uses `--private-address` with a Hetzner/cloud private network that is actually a shared L2 segment, or accidentally supplies a public IP (the registration SSRF guard at nodes.rs:379-482 explicitly ALLOWS public IPs). Container deploy commands, exec streams, and S3 credentials then flow in cleartext, and the agent's API is reachable by anyone who can route to 0.0.0.0:3100. A neighbor on the segment or anyone who can reach the port captures the bearer token and takes over the worker.
- **Recommendation:** For direct mode, require TLS to the agent (self-signed cert fingerprint exchanged during register, pinned by the deployer) and bind the agent to the specific private address rather than 0.0.0.0. Warn loudly (or refuse) when `--private-address` resolves to a public/routable IP without TLS.

### `transport-3` — 🟠 HIGH — Relay (api.temps.sh) is fully trusted TOFU for control-plane identity AND mints the worker's auth token

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-cli/src/commands/join.rs:231-257 (RelayJoinResponse: control_plane_wg_pubkey, control_plane_endpoint, control_plane_url, agent_token); join.rs:252-257 (add_peer with relay-provided pubkey/endpoint); join.rs:286,325 (token: relay_response.agent_token)`
- **What's wrong:** In relay mode the worker accepts, with no pinning or out-of-band verification, the relay's claims about `control_plane_wg_pubkey`, `control_plane_endpoint`, `control_plane_ip`, and `control_plane_url`, then installs that pubkey/endpoint as its WireGuard peer (join.rs:252-257) and sends its registration to that URL. Critically, the *agent_token* itself is supplied by the relay (join.rs:238, used at 286 and persisted at 325) rather than generated locally as in direct mode (join.rs:121). So the relay learns the worker's long-lived control-plane auth token and dictates which WG key / which control plane the worker trusts. There is no relay certificate pinning beyond default web-PKI on the https://api.temps.sh URL, and no signature from the operator's control plane proving the relay relayed honest data.
- **Scenario:** A compromised or malicious relay (or anyone who can impersonate api.temps.sh via a fraudulently issued web-PKI cert) returns its own WG pubkey/endpoint and a control_plane_url it controls. The worker establishes a tunnel to the attacker, registers there, and runs attacker-issued deploy/exec commands — full remote node takeover. Even a passive relay knows every worker's agent_token and can replay it to the real control plane to impersonate the worker (e.g., poison route/peer sync or heartbeat reconciliation).
- **Recommendation:** Have the worker generate its own agent_token locally in relay mode too (never let the relay choose it). Pin the control plane's WG public key out-of-band (the join command/token should carry the expected CP pubkey, so the relay only brokers connectivity, not identity). Sign the relay's join response with the operator's cluster key so the worker can verify the CP identity end-to-end rather than trusting the relay TOFU.

### `transport-4` — 🟠 HIGH — Default inter-node overlay (VXLAN) is unencrypted and can ride a public underlay

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** both · **Confidence:** high · **Location:** `crates/temps-network/src/config.rs:101-115 (default Transport::Vxlan{vni:42,port:4789}); config.rs:65-68 (underlay_address "for public-internet deployments it is the routable public IP"); crates/temps-network/src/linux/vxlan.rs (no encryption); crates/temps-deployments/src/handlers/nodes.rs:731-750 (persist_underlay_address from worker-supplied private_address)`
- **What's wrong:** The multi-host overlay defaults to plain VXLAN encapsulation over UDP 4789 (config.rs:106-109) with no encryption layer anywhere in temps-network (grep for wireguard/encrypt in the crate returns nothing). The underlay address it tunnels to is, by the code's own comment, the routable public IP for public-internet deployments (config.rs:67-68). Cross-node container-to-container traffic (the application data plane between services on different workers) is therefore plaintext on the wire whenever nodes are not already inside an encrypted underlay. WireGuard (temps-wireguard) secures the control/registration path in relay mode but is not wired into temps-network's data-plane transport.
- **Scenario:** A two-node cluster across two cloud regions (or any setup where underlay_address is public) sends inter-service traffic — DB queries, internal API calls, session data — as cleartext VXLAN frames over the public internet. An on-path attacker reads or injects application traffic between containers, bypassing all the TLS that protects user-facing edges.
- **Recommendation:** When the underlay is not provably private, require an encrypted transport for the overlay (run VXLAN inside the WireGuard mesh, or add a WireGuard/IPsec underlay option in NetworkConfig and default to it for non-private underlays). At minimum, refuse VXLAN over a public underlay_address without explicit operator acknowledgment.

### `transport-5` — 🟡 MEDIUM — Global insecure_tls toggle disables cert verification on the remote deployer and all server-side clients

- **Verification:** ⚠️ disputed (1/2 — needs human call)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-core/src/tls.rs:23-51 (process-wide AtomicBool); crates/temps-deployer/src/remote.rs:44-47 (.danger_accept_invalid_certs(temps_core::tls::insecure_tls_enabled())); crates/temps-core/src/app_settings.rs:65 (insecure_tls: bool)`
- **What's wrong:** A single runtime setting (`AppSettings.insecure_tls`) flips a process-wide `AtomicBool` that makes `RemoteNodeDeployer` (and every other server-side client that reads `insecure_tls_enabled()`) build a reqwest client with `danger_accept_invalid_certs(true)`. The flag is documented as 'lab/internal only' and the CLI correctly ignores it, but on the server it is a blanket switch: it disables verification for ALL outbound server clients at once, not just the intended internal agent calls, and it persists in the DB so it can be toggled by anyone with SettingsWrite.
- **Scenario:** An operator enables insecure_tls to make a self-signed internal agent work (the intended use), inadvertently disabling certificate verification for every other outbound HTTPS client in the process (webhooks, git fetches, external integrations, relay/CP calls that read the cache). An attacker with a network position then MITMs any of those unrelated TLS connections. Alternatively, an account that gains SettingsWrite flips it to strip verification cluster-wide.
- **Recommendation:** Scope the insecure-TLS opt-in to the specific agent/node connection (e.g., a per-node 'trust self-signed' flag plus pinned fingerprint) instead of a global flag that affects all outbound clients. Gate the toggle behind a stronger control than ordinary SettingsWrite and surface it prominently in the UI/audit log.

### `transport-6` — ⚪ LOW — No connect timeout on control→agent and agent→control HTTP clients (only request timeout)

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** reliability · **Confidence:** high · **Location:** `crates/temps-deployer/src/remote.rs:44-45 (.timeout(300s), no connect_timeout); crates/temps-agent/src/server.rs:149-151 (heartbeat client .timeout(10s)); crates/temps-agent/src/network_sync.rs:113-117; crates/temps-agent/src/route_sync_client.rs:93-95 (60s timeout)`
- **What's wrong:** The remote deployer uses a 300s overall request timeout and no `connect_timeout`. When an agent's address is reachable at the TCP layer but the host black-holes (common after a node dies but the WG/underlay route lingers), a deploy/exec call can hang up to the full 300s before failing. The agent-side clients set request timeouts but also no connect timeout. There is no circuit breaker keyed off the heartbeat 'offline' state to fail fast before dialing a known-dead node.
- **Scenario:** A worker node hard-fails (kernel panic / network black-hole) while its WG route still resolves. A user-triggered deploy targeting that node blocks for up to 300s holding a workflow slot; multiple such calls exhaust the workflow/connection pool and stall unrelated deployments until each individually times out.
- **Recommendation:** Add a short `connect_timeout` (e.g. 5-10s) on RemoteNodeDeployer and the agent-side clients, and short-circuit remote calls when the node's last heartbeat marks it offline so the control plane fails fast instead of waiting on the full request timeout.

### `transport-7` — ⚪ LOW — Agent Swagger UI / OpenAPI served without authentication on the agent's listen address

- **Verification:** ✅ confirmed (2/2 skeptics)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-agent/src/server.rs:111-115 ("Swagger UI — no auth required"; merged after the auth layer); temps-agent/src/lib.rs:368 (default listen_address 0.0.0.0:3100)`
- **What's wrong:** The agent mounts `/swagger-ui` and `/api-docs/openapi.json` outside the bearer-auth layer (the comment at server.rs:111 states 'no auth required so it's accessible for documentation'). Combined with the default bind of `0.0.0.0:3100`, the full internal agent API surface and its schema are disclosed to anyone who can reach the port, even though invoking the endpoints still requires the token.
- **Scenario:** An attacker who can reach the agent port (e.g., direct-mode worker on a shared segment, or any host that can route to 0.0.0.0:3100) enumerates the complete control surface — container exec, image import, service backup/restore, terminal — from the unauthenticated OpenAPI doc, accelerating targeting and confirming the agent is a Temps worker before attempting token capture.
- **Recommendation:** Disable the Swagger UI/OpenAPI endpoint in production builds or place it behind the same bearer-auth layer, and bind the agent to the specific WG/private interface rather than 0.0.0.0 so the management surface isn't exposed on public interfaces.

**✅ Strengths already in place (do NOT regress):**

- Agent API endpoints are uniformly protected by bearer-token middleware (temps-agent/src/server.rs:107 layers require_agent_auth over all /agent routes) using SHA-256 token hashing and a constant-time comparison (temps-agent/src/auth.rs:25-49), mitigating token timing side-channels.
- The CLI deliberately refuses to honor the server-side insecure_tls opt-in for join-token-carrying registration calls, always building strict-TLS clients in both direct and relay modes (join.rs:124-130, 264-270), with the rationale documented; temps-core/src/tls.rs:13-18 reinforces that the flag never applies to CLI binaries.
- Worker→control-plane channels (heartbeat, network peer sync, route sync) all authenticate with the node bearer token and use strict TLS — network_sync.rs:115 explicitly sets danger_accept_invalid_certs(false), and heartbeat/route-sync clients carry bearer_auth (server.rs:218-221, route_sync_client.rs:151,206).
- The control plane verifies the join token (constant-time hash compare) and blocks ALL node registration when no join token is configured (nodes.rs:585-619), preventing rogue enrollment by default.
- Node registration applies an SSRF guard on both private_address and address, rejecting loopback/link-local/cloud-metadata/multicast/documentation ranges before the values are persisted and later used to build agent/health-check URLs (nodes.rs:379-538).
- Node auth tokens are encrypted at rest with EncryptionService before storage and only decrypted on demand to build a remote deployer (nodes.rs:660-668, 1481-1495); heartbeat/S3-cred endpoints re-verify the token hash per request (nodes.rs:815-821, 910-916).
- Edge TLS private keys are delivered to edge nodes via per-sync ephemeral ECIES (X25519 + AES-256-GCM) with forward secrecy and are gated to role=="edge" only, never workers (nodes.rs:1183-1271).
- WireGuard keypairs are generated with pure-Rust x25519-dalek from OsRng and the private key never leaves the worker (temps-wireguard/src/lib.rs:125-136); the saved agent.json config is written 0600 in a 0700 directory (join.rs:79-106).

## Analyst-added findings (beyond the workflow)

The automated completeness critic was rate-limited, so the analyst added the following from direct code review. The workflow referenced S3 credentials only *in passing* (as a downstream consequence of node takeover / cleartext transport) and never raised the distinct **authorization** gap below.

### `analyst-1` — 🟠 HIGH — `get_s3_credentials` has no per-source authorization scoping (cross-tenant secret exfiltration)

- **Verification:** ✅ verified inline by analyst (read `crates/temps-deployments/src/handlers/nodes.rs:897-974`)
- **Category:** security · **Confidence:** high · **Location:** `crates/temps-deployments/src/handlers/nodes.rs:897-974`
- **What's wrong:** The handler extracts the bearer token, loads the node by `node_id`, and verifies the token hash matches *that node* (so the caller authenticates as itself). It then looks up the S3 source **purely by the `s3_source_id` path parameter** and returns the decrypted `access_key_id` + `secret_key` — with **no check that the source is owned by / scheduled on the requesting node**. Any registered node (or anyone holding any single node token) can enumerate `s3_source_id = 1..N` and pull every configured S3 source's plaintext credentials.
- **Scenario:** One compromised worker (or a rogue node enrolled via the shared join token, `enroll-2`) loops `GET /internal/nodes/{self}/s3-credentials/{i}` for `i` in `1..N` and exfiltrates every tenant's object-storage access+secret keys, then reads/overwrites/deletes all backups and S3-backed buckets across the whole platform.
- **Recommendation:** Scope the lookup: only return credentials for S3 sources the node legitimately needs (sources referenced by a backup/restore job currently assigned to that node, or sources explicitly bound to the node's project set). Reject otherwise with 403 and audit-log the denied access. Consider minting short-lived, source-scoped tokens per backup job instead of handing over long-lived root S3 keys.
