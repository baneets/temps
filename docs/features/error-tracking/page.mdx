## Setup & Integration {{ anchor: true, id: 'setup' }}

Get started with error tracking in minutes using your existing Sentry SDK.

### Step 1: Get Your DSN

Each project gets a unique DSN (Data Source Name) for error reporting. The DSN format is:

```
https://<public_key>@<temps_app>/<project_id>
```

**Example DSN format:**
```
https://742ee692155dd1cb2cecd94838bb425d4b5a1184e5b57c0815e9d808a6f8d70d@your-temps-instance.com/1
```

### Get Your DSN

Visit your project's error tracking setup page to view and copy your unique DSN:

```
https://<temps_app>/projects/<project_slug>/errors?tab=setup
```

**Example:**
```
https://app.temps.davidviejo.dev/projects/my-app/errors?tab=setup
```

On this page, you'll find:
- **Your project's unique DSN** - Copy this to use in your Sentry SDK configuration
- Setup instructions for different platforms
- Environment-specific DSNs (if configured)
- Rate limits and configuration options

<Note>
**Important:** Each project has a unique DSN. You must copy your actual DSN from the setup page—the DSN shown in examples above is just a format reference. Your DSN will have a different public key.

**DSN Format:** The DSN consists of:
- **Public Key**: A 64-character hexadecimal string (unique per project)
- **Temps App URL**: Your Temps instance URL
- **Project ID**: Numeric project identifier

The DSN is automatically generated when you first visit the setup page.
</Note>

### Step 3: Configure SDK

Point the Sentry SDK to your Temps instance:

<CodeGroup title="Configure Sentry SDK">
```typescript {{ title: 'React' }}
import * as Sentry from '@sentry/react';

Sentry.init({
  // Copy your DSN from: https://<temps_app>/projects/<project_slug>/errors?tab=setup
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN || 'YOUR_DSN_HERE',
  environment: 'production',
  release: 'my-app@1.0.0',

  // Performance monitoring
  tracesSampleRate: 1.0,

  // Session replay integration
  integrations: [new Sentry.BrowserTracing()],

  beforeSend(event, hint) {
    // Filter sensitive data
    return event;
  }
});
```

```typescript {{ title: 'Next.js' }}
// sentry.client.config.ts
import * as Sentry from '@sentry/nextjs';

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  environment: process.env.NODE_ENV,
  tracesSampleRate: 1.0,
  replaysSessionSampleRate: 0.1,
  replaysOnErrorSampleRate: 1.0,
});
```

```python {{ title: 'Python' }}
import sentry_sdk
import os

# Copy your DSN from: https://<temps_app>/projects/<project_slug>/errors?tab=setup
sentry_sdk.init(
    dsn=os.getenv("SENTRY_DSN", "YOUR_DSN_HERE"),
    environment="production",
    release="my-app@1.0.0",
    traces_sample_rate=1.0,
)
```

```go {{ title: 'Go' }}
package main

import (
    "github.com/getsentry/sentry-go"
    "os"
)

func main() {
    // Copy your DSN from: https://<temps_app>/projects/<project_slug>/errors?tab=setup
    dsn := os.Getenv("SENTRY_DSN")
    if dsn == "" {
        dsn = "YOUR_DSN_HERE"
    }

    err := sentry.Init(sentry.ClientOptions{
        Dsn: dsn,
        Environment: "production",
        Release: "my-app@1.0.0",
        TracesSampleRate: 1.0,
    })
    if err != nil {
        log.Fatalf("sentry.Init: %s", err)
    }
    defer sentry.Flush(2 * time.Second)
}
```
</CodeGroup>

## Error Grouping {{ anchor: true, id: 'error-grouping' }}

Temps automatically groups similar errors together using smart fingerprinting and AI-powered clustering.

### Grouping Strategies

<Properties>
  <Property name="Stack Trace Fingerprinting">
    Groups errors with identical stack traces (default)
  </Property>
  <Property name="Exception Type">
    Groups by exception class/type (TypeError, ReferenceError, etc.)
  </Property>
  <Property name="Error Message">
    Groups by error message pattern (parameterized)
  </Property>
  <Property name="AI Clustering">
    Uses OpenAI embeddings to find semantically similar errors
  </Property>
  <Property name="Custom Fingerprinting">
    Define custom grouping rules via fingerprint tags
  </Property>
</Properties>

### Custom Fingerprinting

Override default grouping with custom fingerprints:

<CodeGroup title="Custom Fingerprinting">
```typescript {{ title: 'JavaScript' }}
Sentry.captureException(error, {
  fingerprint: ['database-error', 'connection-timeout'],
});

// Dynamic fingerprint
Sentry.captureException(error, {
  fingerprint: ['api-error', error.response?.status],
});
```

```python {{ title: 'Python' }}
with sentry_sdk.configure_scope() as scope:
    scope.fingerprint = ['database-error', 'connection-timeout']
    sentry_sdk.capture_exception(error)
```
</CodeGroup>

### Merge & Split Groups

<Row>
  <Col>
    **Merge Groups**

    Combine multiple error groups that should be treated as one:

    ```bash
    temps error-tracking merge \
      --groups error-1,error-2,error-3 \
      --target error-1
    ```
  </Col>
  <Col>
    **Split Groups**

    Separate errors that were incorrectly grouped:

    ```bash
    temps error-tracking split \
      --group error-1 \
      --event-ids evt-123,evt-456
    ```
  </Col>
</Row>

### Source Context (see the code) {{ anchor: true, id: 'source-context' }}

Temps can show the actual source code around each stack frame, not just
`file:line`. How the source gets there depends on the language:

- **JavaScript / TypeScript** — upload **source maps**. The original source
  travels inside the `.map` (`sourcesContent`), so minified production frames
  resolve to readable code automatically. The official Sentry build plugins and
  `sentry-cli` upload to Temps unchanged (Temps injects `SENTRY_URL` /
  `SENTRY_AUTH_TOKEN` for git-based deploys).

- **Go, Rust, Python, and other native languages** — these already report the
  *original* `file:line`, they just don't ship the source. Upload the raw
  source per release and Temps renders the code around each frame.

  Native source context is **opt-in per project** (Settings → General → *Error
  Tracking Source Context*) — application source is only stored when you enable
  it.

The release you upload under **must match the SDK's reported release**
(`SENTRY_RELEASE`) — typically the deployed commit SHA or tag. For image-based
deploys Temps injects `SENTRY_RELEASE` from the deployed image tag/digest, so
key your upload the same way.

<CodeGroup title="Upload source in CI (native)">
```bash {{ title: 'Whole tree' }}
# Upload every source file under the repo, keyed by the release.
# file_path is each file's path relative to the root — exactly how a
# -trimpath Go binary (and most native SDKs) report frame filenames.
bunx @temps-sdk/cli errors source-files upload \
  --project-id 42 \
  --release "$CI_COMMIT_SHA" \
  --dir .
```

```bash {{ title: 'Single file' }}
bunx @temps-sdk/cli errors source-files upload \
  --project-id 42 \
  --release "$CI_COMMIT_SHA" \
  --file internal/gateway/middleware.go \
  --file-path internal/gateway/middleware.go
```

```bash {{ title: 'List / delete' }}
bunx @temps-sdk/cli errors source-files list   --project-id 42 --release "$CI_COMMIT_SHA"
bunx @temps-sdk/cli errors source-files delete --project-id 42 --release "$CI_COMMIT_SHA"
```
</CodeGroup>

## Error Resolution {{ anchor: true, id: 'resolution' }}

Manage error lifecycle from discovery to resolution.

### Error States

<Properties>
  <Property name="Unresolved">
    New or ongoing errors (default state)
  </Property>
  <Property name="Resolved">
    Marked as fixed (manually or automatically)
  </Property>
  <Property name="Ignored">
    Known errors that should be suppressed
  </Property>
  <Property name="Regressed">
    Error recurred after being resolved
  </Property>
</Properties>

### Mark as Resolved

<CodeGroup title="Resolve Errors">
```bash {{ title: 'CLI' }}
# Resolve error group
temps error-tracking resolve --group error-123

# Resolve in specific release
temps error-tracking resolve --group error-123 --in-release 1.0.1

# Resolve multiple groups
temps error-tracking resolve --groups error-123,error-456,error-789
```

```typescript {{ title: 'Dashboard' }}
// Resolve via dashboard
// Navigate to: Error Details > Actions > Resolve

// Options:
// - Mark as resolved now
// - Resolve in next release
// - Resolve in specific release
// - Set auto-resolve (if no new events in X days)
```

```typescript {{ title: 'API' }}
// Resolve error group via API
await fetch('/api/error-groups/error-123/resolve', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${API_KEY}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    resolution: 'next_release',
    comment: 'Fixed in PR #456'
  })
});
```
</CodeGroup>

### Auto-Resolution

Configure automatic resolution rules:

```yaml
# Auto-resolve errors
auto_resolution:
  # Resolve if no new events for X days
  no_events_for: 30

  # Resolve when new release is deployed
  on_release: true

  # Resolve when error rate drops below threshold
  rate_threshold:
    errors_per_hour: 1
    duration: 24h
```

### Ignore Errors

Suppress known errors you don't want to track:

<CodeGroup title="Ignore Errors">
```typescript {{ title: 'Client-Side' }}
Sentry.init({
  dsn: '...',
  ignoreErrors: [
    // Ignore specific error messages
    'ResizeObserver loop limit exceeded',
    'Non-Error promise rejection',

    // Ignore error patterns
    /^Loading chunk [\d]+ failed$/,
  ],

  // Ignore errors from specific URLs
  denyUrls: [
    /extensions\//i,
    /^chrome:\/\//i,
  ],

  beforeSend(event, hint) {
    // Custom ignore logic
    if (event.exception) {
      const error = hint.originalException;
      if (error.message?.includes('Network error')) {
        return null; // Don't send to Temps
      }
    }
    return event;
  }
});
```

```bash {{ title: 'CLI' }}
# Ignore error group
temps error-tracking ignore --group error-123

# Ignore with pattern
temps error-tracking ignore --pattern "Loading chunk .* failed"
```
</CodeGroup>
